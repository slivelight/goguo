## 1. 问题陈述

Feature 001（OPP-002）、Feature 002（OPP-004）、Feature 003（OPP-001）已定义 GoGuo 的全部后端能力——baseline 评估与恢复、WSL/Linux 侧配置、目标站点规则与可达性诊断。但三个 feature 均通过约束（Feature 001 CON-3、Feature 002 排除项、Feature 003 OP-5）明确将用户交互界面排除在本轮范围之外。

当前核心矛盾：三个 feature 定义的所有能力（baseline 确认、站点管理、规则预览、诊断提示、恢复通知等）都需要用户交互界面来触发和呈现。缺少 UI 层意味着：

1. 用户无法完成 baseline 确认流程（Feature 001 的前提条件）
2. 用户无法管理目标站点、选择预设模板（Feature 003 的核心入口）
3. 用户无法看到可达性诊断结果和恢复进度（Feature 003 的产品价值呈现）
4. 用户无法在规则应用前预览和确认（Feature 001/003 的安全要求）
5. Windows 与 WSL/Linux 两侧状态差异不可见（Feature 002 的诊断价值）

本轮问题不是"构建完整的通用代理客户端 GUI"，而是回答：GoGuo 的最小交互界面是什么，才能让 Feature 001/002/003 定义的全部能力被用户触达。

## 2. 目标用户与使用情境

| 用户 | 使用情境 | Struggling Moment | 期望进展 |
|------|----------|-------------------|----------|
| 普通办公用户 | 安装 GoGuo 后首次使用 | 不知道 GoGuo 是否在工作、当前网络状态如何、目标站点是否可达 | 一眼看到状态，无需理解技术细节 |
| PC 端开发者 | 添加目标站点、切换网络后检查恢复状态 | 需要通过命令行或配置文件管理站点和规则，操作门槛高 | 可视化管理站点、实时查看诊断结果 |
| 进阶用户 | 查看规则详情、覆盖自动生成的规则 | 无法预览生成的规则，不清楚哪些流量走代理哪些直连 | 透明展示规则，支持必要的自定义 |

本轮主用户是**普通办公用户**和**PC 端开发者**。进阶用户的深度自定义需求作为后续迭代输入。

## 3. Why now / 当前价值判断

`OPP-003` 从 P1 升级为阻塞依赖，原因：

- **技术栈调整**：项目技术栈发生调整，原 BAT/PowerShell 交互方式不再满足需求。
- **三个 feature 的能力缺口**：Feature 001 需要用户确认 baseline、Feature 003 需要站点管理和规则预览——这些交互需求在三个 feature 的规格中通过"交互语义"描述但未实现，必须由 OPP-003 统一承载。
- **产品闭环要求**：用户安装 GoGuo 的根本动机是让目标站点可用。没有 UI，用户无法完成从安装→配置→诊断→恢复的完整闭环。
- **战略发现原始判断**：OPP-003 被定位为"最小状态面板"，当时的假设是"先做 Web/托盘壳比 Tauri 全量重构更快"。但用户明确要求独立桌面应用，且实际需求已超出"状态面板"——站点管理、规则预览、诊断提示等交互操作也需要完整 UI 支持。

因此，本轮 discovery 的范围从"最小状态面板"扩展为"独立桌面应用"（Minimum Viable Desktop Application），在不同平台下提供一致的操作界面，但仍然不追求通用代理客户端的全部 GUI 功能。

## 4. 当前轮 wedge / 最小机会点

当前轮唯一主 wedge：

> 独立桌面应用：提供 GoGuo 服务控制、站点管理、状态监控、诊断展示和规则预览的跨平台桌面应用，在不同平台下保持一致的程序操作界面，使 Feature 001/002/003 定义的全部能力可被用户触达。

本轮将"用户可完成完整操作闭环"作为完成标准。但不追求自定义主题、多语言、完整设置面板、审计日志浏览器等高级功能。

## 9. 成功度量

- Desired Outcome：用户通过 UI 完成 GoGuo 的完整操作闭环（安装→配置→监控→诊断→恢复），无需命令行或手动编辑配置文件。
- North Star 锚定：战略发现 Opportunity C "普通用户与开发者可理解的状态面板"。
- Leading 指标：
  - 首次安装后 baseline 确认完成率。
  - 站点添加到规则生效的操作步骤数。
  - 用户无需帮助文档即可完成核心操作的比例。
- Lagging 指标：
  - 用户因 UI 不可用而绕过 UI 直接操作配置文件的比例。
- Success Threshold：
  - 主路径操作不超过 2 步（战略发现 Success Threshold）。
  - 状态显示与后端实际状态一致。
  - 失败提示可触发手动处理。
  - 全部通知（规则回退、恢复进度）可在 UI 中查看。
  - Windows 和 WSL/Linux 两侧均可访问 UI。
- Non-goal Metrics：
  - 不追求 UI 美观度评分。
  - 不追求自定义主题/插件。
  - 不追求与通用代理客户端 GUI 功能完全对等。

  - Assumption：系统通知在目标平台上可用。
  - Probe：通知机制验证。

| 问题 | 为什么阻塞 | 建议处理 |
|------|------------|----------|
| UI 技术选型（独立桌面应用框架：Tauri/Electron/其他） | 技术选型决定架构方案、跨平台一致性和开发成本 | hf-design 阶段确定 |
| 后端 API 覆盖度 | 技术栈调整后 API 层可能需要重建 | P0 API 覆盖度评估 |

### 非阻塞项

| 问题 | 说明 |
|------|------|
| UI 入口方式（桌面快捷方式/托盘/启动器） | 用户习惯考量 |
| WSL/Linux 侧通知替代方案 | WSL 通知能力受限时的降级策略 |
| 离线状态下的 UI 行为 | 后端未运行时 UI 如何展示 |
