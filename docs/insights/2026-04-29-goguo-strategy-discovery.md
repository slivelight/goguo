## 1. 执行摘要

| 项目 | 内容 |
|------|------|
| 业务方向 | GoGuo 是面向普通办公用户的本地网络可达性工具，在国内网络环境下跨家庭、办公、差旅网络切换时，让被阻断的目标网站自动保持可用，同时不破坏原本可直连网站的访问。 |
| 当前核心矛盾 | 既有 github-host 原型具备智能代理编排、节点池、IP 池、监控恢复等强能力，但用户体验交付层仍停留在 BAT/PowerShell，且 proxy/direct/hosts 是整机级二选一，导致“修一个站点，伤害其他站点”。 |
| 候选战略假设 | 以“PC 端 Windows + Linux/WSL 开发环境支持、按站点/按网络自动决策、可恢复到已确认网络基线、用户无感监控”为候选控制点，而不是做一个完整通用代理客户端。该方向仍需通过 blocking probe 后才能锁定。 |
| 年度目标 | 在 PC 端 Windows + Linux/WSL 形态下验证“安装后网络评估与基线确认 + 一键可达 + 网络切换自动恢复 + 失败可解释”的产品闭环，再决定是否进入移动端和自建服务。 |

本轮战略建议: GoGuo 不应从“更完整的 Clash Verge Rev”切入，而应从“PC 端网络可达性诊断、基线恢复与目标站点规则辅助”切入。当前 MVP 范围应覆盖 Windows 与 Linux/WSL 结合的开发者工作流，包括技术网站浏览、VS Code 开发、agent 工具订阅、GitHub 项目管理与 wiki 撰写；移动端暂不进入当前轮。现有 github-host 的领先能力在智能编排层，不在 UI、跨平台或协议广度；第一轮产品规划应先证明“安装后网络评估可形成可信基线、系统改动可恢复、非目标站点不误伤、目标站点可解释地恢复可用”。在这些假设未验证前，不公开承诺“被墙网站自动可用”，也不把自建海外服务放入 MVP。

### JTBD Jobs Stories

- When 我从家里切到公司或酒店网络，I want GoGuo 自动重新选择访问策略，so I can 继续打开常用目标网站而不需要理解代理配置。
- When 我在 Windows + WSL/Linux 环境中同时使用浏览器、VS Code、agent 工具、GitHub 和 wiki，I want GoGuo 让这些入口共享一致的目标站点可达性基线，so I can 不在 Windows 与 Linux 子系统之间反复排查代理差异。
- When 我只想访问某几个目标网站，I want 工具只处理这些目标网站的流量，so I can 不影响其他本来能直连的网站。
- When 自动恢复失败，I want 收到明确原因、下一步建议和一键恢复到已确认网络基线的能力，so I can 不把整台电脑和开发环境网络弄坏。
- When 我是进阶用户，I want 能查看当前站点、节点、模式和日志，so I can 在需要时手动接管。

**差异化策略**: GoGuo 的差异化不是“协议更多、平台更多、GUI 更完整”，而是“PC 端目标站点维度的自动可达性编排 + 已确认网络基线恢复”。安装后先做网络评估，协助用户调整到可接受状态，并将该状态记录为初始网络基线；后续网络切换、自动修复、停止服务和失败恢复都以这个已确认基线为参照。具体表现为：按站点选择 direct/hosts/proxy，网络变化后自动重评估，失败时有恢复链和用户可理解的状态，停止后能把 Windows 与 Linux/WSL 相关网络配置恢复到基线。

### OST Snapshot

- Desired Outcome: PC 端普通办公用户与开发者跨网络切换后，指定目标网站和开发工具链持续可用，且不影响原本可直连的网站。
- Safety Prerequisite: 自动切换系统网络状态前，必须先完成安装后网络评估，形成用户确认的 baseline，并证明 baseline/restore、Proxy Guard、审计和二次确认可靠。
- Opportunity A: 站点级策略选择，解决整机 proxy/direct/hosts 二选一导致的误伤。
  - Solution A1: 基于 mihomo 标准规则能力、sites 配置、网络指纹和可达性探测，生成目标站点规则与诊断建议。
  - Assumption: 现有 mihomo rule-provider、sites 配置、DecisionEngine 能支持站点级策略落地，且不会污染非目标站点直连。
- Opportunity B: 安装后网络评估与基线恢复。
  - Solution B1: 安装后先进行 Windows + Linux/WSL 网络评估，协助用户调整到可接受状态，将该状态保存为初始网络基线；后续停止、失败恢复和自动修复均回到该基线，而不是回到未经确认的“系统原状”。
  - Assumption: Windows 与 Linux/WSL 侧关键网络状态都能可靠读取、原子保存和回滚。
- Opportunity C: 普通用户与开发者可理解的状态面板。
  - Solution C1: 基于 MonitorServer `:9091` 提供最小 Web UI/托盘状态，而不是先重写 Tauri。
  - Assumption: 现有 REST API 足够支撑 P0 状态与操作闭环。

## 7. 三定决策

### 7.1 定战略控制点（7 Powers）

| 控制点 | 描述 | Power类型 | 可持续性 | 依据来源 |
|--------|------|----------|----------|----------|
| PC 端基线恢复闭环 | 安装后先完成 Windows + Linux/WSL 网络评估并形成用户确认的基线；所有系统改动可审计、可二次确认、可恢复到该基线，失败时给出可理解建议。自动切换能力必须以此为前置门槛。 | 品牌 / 转换成本 | 中 | `01-index.md` 问题清单；AGENTS 安全声明；知识库第17/18/22章；用户评审反馈 |
| 站点级规则诊断与自动配置 | 基于目标网站、网络环境、可达性探测结果生成 mihomo 标准规则与诊断建议，而不是自称替代通用代理规则能力。 | 流程优势 / 反定位 | 中 | `01-index.md` 致命问题；`arch-AA.md` DecisionEngine；知识库第8章；subagent challenger finding；用户评审反馈 |
| 节点/IP/订阅质量数据闭环 | 通过节点池、IP 池、订阅源质量追踪形成持续可用性优势。 | 流程优势 / 垄断资源 | 中 | `02-github-host产品gap分析.md` 订阅管理、节点管理领先 |

**选择理由**:
- 通用代理客户端已在 GUI、跨平台、多规则方面领先，GoGuo 不应完整复制；但应兼容 mihomo 标准配置、订阅、rule provider 和 PC 端部署生态。
- 知识库第8章说明“站点级分流”本身是代理客户端标准能力，因此 GoGuo 的控制点不能写成“自研站点级网络能力”，而应写成“规则配置、可达性诊断、恢复闭环和失败解释”的流程优势。
- 现有原型最独特的是“检测、决策、恢复、节点/IP 生命周期”组合能力，应该被产品化为普通用户和 PC 端开发者可理解的本地网络可达性辅助体验。
- 恢复能力不能只回到未经确认的系统原状，而应回到安装后网络评估形成的可用基线；未验证前不得放开自动切换系统网络状态。

## 8. Multi-Agent 辩论记录

### 8.1 辩论问题

GoGuo 第一阶段是否应优先做“PC 端 Windows + Linux/WSL 的站点级可达性 + 基线恢复闭环”，而不是先做完整通用代理客户端、移动端客户端或自建海外服务？

### 8.2 正方论证（Proponent）

| 核心论据 | 依据 | 强度 |
|----------|------|------|
| 当前最大用户失败不是协议数量不足，而是全局 proxy/direct/hosts 模式误伤非目标站点。 | `01-index.md` 致命问题；知识库第8章“分流规则”说明全局代理会造成浪费、延迟和异常。 | 8/10 |
| 现有资产强在智能编排，不强在通用客户端 GUI/跨平台。 | `02-github-host产品gap分析.md` 显示 DecisionEngine、三路探测、节点池、IP 池、订阅质量、DNS 污染检测、HA 是独有优势。 | 8/10 |
| 本地可信恢复是普通用户信任前提。 | `01-index.md` 指出 Stop All Modes + Monitor 后仍可能不能恢复；AGENTS 要求敏感操作二次确认 + 审计日志。 | 9/10 |
| 移动端不是当前 wedge 必要条件，但 PC 端 Linux/WSL 支持是目标用户工作流的一部分。 | gap 分析把跨平台标为 Phase 5 目标下的 Critical；用户评审反馈明确当前开发环境需要 Windows + WSL 协同。 | 8/10 |
| 自建海外服务会提前引入合规、运维、隐私和成本风险。 | 知识库第10章涉及 VPS/IP/线路运维；第17章指出日志隐私风险；第22章提示提供或销售服务的合规风险。 | 8/10 |

正方建议: GoGuo MVP 不完整复制通用代理客户端、不先做移动端、不先绑定自建海外服务。先在 PC 端 Windows + Linux/WSL 环境证明目标站点可用、非目标站点不受损、停止后恢复到已确认基线。

### 8.3 反方质疑（Challenger）

| 核心质疑 | 依据 | 强度 |
|----------|------|------|
| 草稿把 blocking 假设写成确定结论，证据状态冲突。 | 原草稿同时列出用户接受度、站点级策略、baseline/restore、合规表达为 blocking 假设。 | 9/10 |
| 市场定位证据不足。 | 当前原型材料主要证明 GitHub/多站点 Windows 工具，不证明“普通办公用户 85%”大市场；用户评审反馈补充了 PC 端开发者/知识工作者这一更具体目标用户。 | 8/10 |
| 站点级分流不是足够强的独占差异化。 | 知识库第8章说明 DOMAIN/IP-CIDR/GEOIP/PROCESS/RULE-SET 是代理客户端标准能力；现有项目规则系统仍缺 IP-CIDR/GEOIP/RULE-SET/provider。 | 9/10 |
| 技术可行性被低估。 | `arch.md` 显示 ChatGPT、Claude、Google、npmjs 的 hosts 模式基本不可用，只有 Proxy 模式覆盖全部 5 站点；若依赖 mihomo，则站点级本质是标准代理规则能力。 | 8/10 |
| 合规风险不是文案问题。 | 知识库 `index.md` 和第22章强调合法合规边界，未经许可提供服务、销售或大规模组织相关服务风险高。 | 9/10 |

反方建议: GoGuo 当前不应定义为“普通用户的自动翻墙工具”，也不应公开承诺“被墙网站自动可用”。先作为 PC 端本地网络可达性诊断与恢复工具，验证安装后基线、目标站点规则配置、系统改动可回滚、Windows + Linux/WSL 一致可用、失败可解释五个能力。

### 8.4 仲裁综合（Arbiter）

| 争议点 | 仲裁结论 | 置信度 |
|--------|----------|--------|
| 是否坚持“站点级自动可达性” | 有条件采纳。它是真实用户痛点，但不能写成确定战略，也不能伪装成独占网络能力；应表述为“目标站点规则配置与可达性诊断层”。 | 75% |
| 是否先做恢复闭环 | 采纳反方并吸收用户评审反馈。安装后网络评估、baseline/restore、Proxy Guard、二次确认和本地审计必须先于自动切换系统网络状态。 | 90% |
| 是否先做通用代理客户端 | 不完整采纳。通用代理客户端竞争强，GoGuo 当前资源和资产不匹配；但必须兼容 mihomo 标准配置、订阅、rule provider 和 PC 端部署生态。 | 85% |
| 是否先做跨平台 | 修正为“先做 PC 端跨 Windows + Linux/WSL，不做移动端”。Linux/WSL 支持进入 MVP 或紧邻 MVP 的 P0/P1 范围。 | 90% |
| 是否先做自建海外服务 | 不采纳。服务端会引入合规、日志、运维、IP 质量和成本风险，应降级为 P2 假设。 | 85% |

**最终决策建议**: 有条件采纳。GoGuo 当前战略应改写为“PC 端 Windows + Linux/WSL 网络可达性诊断、基线恢复与目标站点规则辅助工具”。OPP-002 是安全前置门槛，OPP-004 是目标用户环境前置适配，OPP-001 是产品价值 wedge；三者都必须通过 probe 才能进入正式规格。
**置信度**: 80%

| OPP-003 | 最小状态面板：普通用户与开发者能看懂当前是否可用、正在修复什么、相对 baseline 偏离在哪里。 | 用户不用打开 BAT 菜单，也能看到目标站点、当前策略、网络状态、恢复进度和 Windows/WSL 差异。 | 状态显示与 MonitorServer 一致；主路径不超过 2 个操作；失败提示可触发手动处理。 | `:9091` API 足够支撑最小 UI；先做 Web/托盘壳比 Tauri 全量重构更快。 | P1 | 否 |
| OPP-005 | 节点/订阅质量产品化：把已有领先能力变成稳定性优势。 | 工具能自动选择更可靠节点/订阅源，并解释为什么切换。 | 节点可用率、来源成功率、失败原因可查询；自动切换不频繁抖动。 | 现有节点池数据质量足够；普通用户不需要直接管理复杂节点。 | P1 | 否 |
| OPP-006 | 自建海外服务与移动端演进。 | 在 PC 端 MVP 验证后，为稳定性和商业化提供托管服务或移动端客户端。 | 有明确留存、失败率、付费意愿证据后再立项。 | PC 端 MVP 指标能证明需求；合规与成本可接受。 | P2 | 否 |

- 候选战略方向: GoGuo 优先验证“PC 端 Windows + Linux/WSL 网络可达性诊断、基线恢复与目标站点规则辅助工具”，而不是完整通用代理客户端。
- 候选战略控制点: PC 端基线恢复闭环、Linux/WSL 开发入口诊断、目标站点规则诊断与自动配置、节点/IP/订阅质量数据闭环。
- 候选 Desired Outcome: 跨家庭、办公、差旅网络切换时，指定目标网站和开发工具链可解释地恢复可用，且不影响原本可直连网站。
- 候选 Success Threshold: OPP-002 先证明安装后 baseline 与系统改动可恢复；OPP-004 证明 Windows + Linux/WSL 入口一致可诊断；OPP-001 再证明 P95 恢复 <= 10s、非目标直连不低于 baseline、用户无需手动理解模式。
- 需要继续保留为 assumption 的内容:
  - 用户是否接受“PC 端 Windows + Linux/WSL 诊断与目标站点辅助”的 MVP 范围。
  - 当前代码是否能以低风险方式从 Windows 全局模式调整到站点级策略，并为 Linux/WSL 提供足够诊断/配置提示。
  - 国内网络与组织网络策略下的合规边界如何表达。
  - Web/托盘最小 UI 是否足以替代 BAT 主路径。
- 当前不进入功能收敛的候选项:
  - 自建海外 Oracle Cloud 服务。
  - 移动端客户端。
  - TUIC/WireGuard 协议补齐。
  - 通用规则市场、WebDav 同步、完整 Tauri 重构。

**多 Opportunity 说明**:
- OPP-002 作为安全前置 wedge，优先进入 hf-product-discovery。
- OPP-004 作为目标用户环境前置适配，与 OPP-002 一起约束 OPP-001。
- OPP-001 作为产品价值 wedge，依赖 OPP-002 的恢复和审计前置门槛，也依赖 OPP-004 的 PC 端 Windows + Linux/WSL 诊断能力。
- 每个 Opportunity 独立走: hf-product-discovery → hf-discovery-review → hf-specify。
