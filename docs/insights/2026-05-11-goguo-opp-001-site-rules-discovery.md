## 1. 问题陈述

Feature 001（OPP-002）已证明 GoGuo 可以安全地评估、确认和恢复到用户 baseline。Feature 002（OPP-004）已将此能力扩展到 WSL/Linux 侧。但安全底座本身不产生产品价值——用户安装 GoGuo 是为了让被阻断的目标网站可用，而不是为了"能安全地恢复"。

当前核心矛盾：现有原型采用整机级 proxy/direct/hosts 二选一模式，导致"修一个站点，伤害其他站点"的致命问题。用户只需要少数目标网站（GitHub、Google、Stack Overflow、Claude 等）可用，而不需要所有流量走代理。

本轮问题不是"构建完整的通用代理规则引擎"，而是回答：GoGuo 能否基于目标站点列表，自动生成合适的访问策略（直连/代理），在 Windows 和 WSL/Linux 两侧让目标站点可解释地恢复可用，同时确保非目标站点的可达性不低于 baseline。

## 2. 目标用户与使用情境

| 用户 | 使用情境 | Struggling Moment | 期望进展 |
|------|----------|-------------------|----------|
| 普通办公用户 | 需要访问 Google 搜索、Gmail、Google Docs 等 3-5 个目标站点 | 开启代理后其他国内网站变慢或不可用；关闭代理后目标站点不可用 | 只处理目标站点，其他网站不受影响，无需理解代理/直连概念 |
| PC 端开发者 | 需要 GitHub、npm registry、Stack Overflow、Claude API、Docker Hub 等 | 全局代理模式下 npm 镜像源访问异常、公司内部网站不可达 | 目标开发站点自动走代理，内部站点和镜像源走直连 |
| PC 端知识工作者 | 需要 Wikipedia、Medium、学术搜索等技术文档站点 | 不知道哪些站点需要代理、哪些直连更快 | 自动判断并选择最快路径 |

本轮主用户是普通办公用户和 PC 端开发者。知识工作者和进阶用户的站点级调优需求作为后续迭代输入。

## 3. Why now / 当前价值判断

`OPP-001` 是产品价值 wedge，原因：

- OPP-002（Feature 001）已提供 baseline/restore 安全底座，自动切换网络状态的前提条件已满足。
- OPP-004（Feature 002）已提供 WSL/Linux 侧配置能力，目标站点可达性可同时在两侧生效。
- 现有原型已有 DecisionEngine、sites 配置、mihomo 集成等可复用基础，但缺乏站点级规则的产品化封装。
- "整机 proxy/direct 二选一误伤非目标站点"是战略发现确认的致命问题，必须在 MVP 中解决。
- 用户安装 GoGuo 的根本动机是让目标站点可用；仅提供安全恢复不足以形成产品闭环。

因此，本轮 discovery 不追求完整的规则引擎、自定义分流或可视化规则编辑器，而是先证明 GoGuo 可以基于一个目标站点列表，自动生成合适的访问策略，并在 Windows 和 WSL/Linux 两侧一致生效。

## 4. 当前轮 wedge / 最小机会点

当前轮唯一主 wedge：

> 目标站点规则配置与可达性诊断：基于用户配置的目标站点列表，GoGuo 自动判断每个目标站点的访问策略（直连/代理），生成对应的 mihomo 规则配置，在 Windows 和 WSL/Linux 两侧生效，同时持续诊断目标站点可达性并在不可达时触发恢复或提示。

本轮将"自动让目标站点持续可用"作为完成标准。但不追求自定义规则编辑、规则市场、用户自定义分流逻辑等高级功能。

## 6. 关键假设与风险

### Desirability

| 假设 | 风险 | 本轮处理 |
|------|------|----------|
| 普通办公用户接受"告诉 GoGuo 你需要哪些站点"的配置方式 | 若配置流程过重，用户可能直接放弃 | 提供预设站点模板（如"开发者套装"），降低配置门槛 |
| 开发者接受 GoGuo 自动生成规则而非手动编辑规则 | 若规则不符合预期，开发者会绕过 GoGuo | 提供规则预览和覆盖机制，保持透明度 |

### Feasibility

| 假设 | 风险 | 本轮处理 |
|------|------|----------|
| mihomo rule-provider 能支持 GoGuo 的站点级策略需求 | 社区规则集覆盖度和更新频率不确定 | 本轮使用最小规则集（仅目标站点），不依赖完整社区规则 |
| 目标站点集合可先限制在少数高价值站点 | 若用户需求超出少数站点，规则复杂度会上升 | 本轮限制为 10-20 个目标站点，验证核心流程 |
| 现有 DecisionEngine 探测能力可复用于站点可达性诊断 | 探测机制可能需要适配新场景 | 设计阶段评估复用范围 |
| Windows 和 WSL/Linux 两侧可通过同一 mihomo 实例或配置同步实现规则生效 | WSL/Linux 侧可能需要独立的代理配置 | 设计阶段确定架构方案 |

### Usability

| 假设 | 风险 | 本轮处理 |
|------|------|----------|
| 用户不需要理解 DOMAIN/DOMAIN-SUFFIX/IP-CIDR 等规则类型 | 若规则类型暴露给用户，会增加认知负担 | 规则生成完全自动，用户只需指定站点域名 |

### Viability / 合规

| 假设 | 风险 | 本轮处理 |
|------|------|----------|
| 底层代理行为不会污染非目标直连站点的访问 | 若 mihomo 配置不当导致全局代理，非目标站点会受损 | 默认策略为"DIRECT 兜底 + 目标站点 PROXY"，确保非目标站点不受影响 |

## 9. 成功度量

- Desired Outcome：用户跨网络切换后，目标网站可解释地恢复可用，非目标网站不受影响。
- North Star 锚定：战略发现 Objective 1 KR1.2 "目标站点在网络切换后自动恢复可用" 和 KR1.3 "不误伤非目标直连网站"。
- Leading 指标：
  - 目标站点规则生成成功率。
  - 目标站点在网络切换后恢复可用的耗时。
  - 非目标直连站点可达性偏离 baseline 的比例。
- Lagging 指标：
  - 用户因非目标站点受影响而停止使用 GoGuo 的比例。
- Success Threshold：
  - 目标站点在网络切换后 MVP 阶段 P95 恢复 <= 30s。
  - 非目标直连站点可达性不低于 baseline。
  - 规则策略为"默认 DIRECT + 目标 PROXY"，非目标站点不经过代理。
  - 规则生成后 Windows 与 WSL/Linux 两侧目标站点均可访问。
  - 目标站点不可达时有明确的诊断提示。
- Non-goal Metrics：
  - 不追求 P95 <= 10s（留给后续优化）。
  - 不追求自定义分流逻辑。
  - 不追求流量统计面板。

## 10. JTBD 视图

### Jobs Stories

- When 我只想访问 GitHub 和 Google，I want GoGuo 只处理这两个站点的流量，so I can 国内网站访问速度不受影响。
- When 我从家里切到公司网络，I want GoGuo 自动重新评估并让目标站点恢复可用，so I can 不需要手动切换模式或重新配置。
- When 目标站点突然不可用，I want GoGuo 告诉我是网络问题、节点问题还是规则问题，so I can 知道下一步怎么处理。

### 四力分析

| 四力 | 本轮表现 |
|------|----------|
| Push of the situation | 全局代理模式误伤非目标站点是致命问题，用户不得不频繁手动切换。 |
| Pull of the new solution | 只处理目标站点、自动判断策略、网络切换后自动恢复。 |
| Anxiety of the new solution | 担心规则配置不当导致所有站点不可用或隐私泄露。 |
| Habit of the present | 用户继续手动切换全局代理开关，或使用其他代理客户端手动配置分流规则。 |

## 11. OST Snapshot

Desired Outcome：用户跨网络切换后目标站点可解释地恢复可用，非目标站点不受影响。

Opportunity A：站点级策略选择，解决整机二选一误伤。

- Solution A1：基于目标站点列表自动生成 mihomo 规则。
  - Assumption：mihomo 标准规则能力足够支持 GoGuo 的站点级策略。
  - Probe：最小规则集验证。
- Solution A2：默认直连策略，仅目标站点走代理。
  - Assumption："DIRECT 兜底 + 目标 PROXY"策略不会导致非目标站点受损。
  - Probe：非目标站点可达性对比测试。

Opportunity B：目标站点可达性持续诊断。

- Solution B1：定期探测目标站点可达性，不可达时触发恢复或告警。
  - Assumption：DecisionEngine 探测能力可复用。
  - Probe：决策引擎复用评估。

## 12. Bridge to Spec

若本 discovery 通过 `hf-discovery-review`，建议进入 `hf-specify` 的范围如下。

### 推荐进入规格的范围边界

- 目标站点列表管理（添加、删除、预设模板）。
- 基于目标站点列表自动生成 mihomo 规则配置（DOMAIN/DOMAIN-SUFFIX 规则类型）。
- 默认直连策略："DIRECT 兜底 + 目标站点 PROXY"。
- 站点可达性持续诊断（探测间隔、探测方式、探测结果记录）。
- 目标站点不可达时的恢复动作或告警提示。
- 规则预览和覆盖机制。
- Windows 与 WSL/Linux 两侧规则同步。
- 规则变更时的 baseline 差异检查（确保规则变更不破坏 baseline 承诺）。

### 可直接转成规格输入的稳定结论

- 规则策略必须是"默认 DIRECT + 目标 PROXY"，不采用全局代理模式。
- 安全前置条件已由 Feature 001 和 Feature 002 满足。
- mihomo 规则能力是标准能力，GoGuo 的差异化在于自动生成和可达性诊断。
- 目标站点集合先限制为少数高价值站点。
- 规则变更必须记入 Feature 001 定义的统一审计流。

### 需要继续保留为 assumption 的内容

| Assumption | 初始置信度 | 置信度来源 | 后续验证 |
|------------|------------|------------|----------|
| mihomo rule-provider 能支持 GoGuo 的最小规则集需求 | 中 | mihomo 文档确认标准能力，但与 GoGuo 集成未经验证 | P0 最小规则集 probe |
| DecisionEngine 探测能力可复用于站点可达性诊断 | 中 | 现有原型已有探测能力，但适配范围未评估 | P1 决策引擎复用评估 |
| 10-20 个目标站点的规则集不会显著影响 mihomo 性能 | 高 | 规则集规模小，性能影响预计可忽略 | 规则生成后性能监控 |
| 默认 DIRECT 策略下非目标站点确实不受影响 | 高 | 这是 mihomo 的基本行为，但需验证配置正确性 | P0 非目标站点对比测试 |

### 当前不进入 spec 的候选项

- 自定义规则编辑器
- 规则市场/社区规则订阅
- 用户自定义分流逻辑（按进程、按时间段）
- 完整流量统计和面板
- 节点选择策略优化

### 带入规格阶段的成功标准锚点

- 目标站点 MVP 阶段 P95 恢复 <= 30s。
- 非目标直连站点可达性不低于 baseline。
- 默认 DIRECT + 目标 PROXY 策略。
- Windows 与 WSL/Linux 两侧目标站点均可访问。
- 不可达时有五要素诊断提示。
