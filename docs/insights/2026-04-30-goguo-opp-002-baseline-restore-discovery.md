## 1. 问题陈述

GoGuo 的核心价值依赖自动修改本机网络状态：系统代理、hosts、TUN、代理进程、DNS 缓存，以及 Windows 与 WSL/Linux 协同环境中的代理感知状态。但现有原型已经暴露一个信任问题：用户停止工具后，电脑不一定回到可用网络状态，甚至可能影响原本无需配置即可访问的网站。

这会阻塞普通办公用户和 PC 端开发者采用 GoGuo。对这类用户来说，最重要的不是工具是否展示了复杂模式，而是“我能不能随时停用它，电脑和开发环境能不能回到我确认过的可用状态”。如果这个安全前置不成立，后续的自动切换、站点级规则、WSL/Linux 协同环境支持和 UI 包装都没有稳定信任基础。

本轮问题不是“实现一个完整网络配置管理系统”，而是先回答：GoGuo 能否在安装后完成网络评估，形成用户确认的 baseline，并在停止、失败恢复、代理异常或网络切换后，可靠地恢复到该 baseline，或者明确说明哪些项无法自动恢复、需要用户处理。

## 2. 目标用户与使用情境

| 用户 | 使用情境 | Struggling Moment | 期望进展 |
|------|----------|-------------------|----------|
| 普通办公用户 | 在家庭、公司、酒店、机场、移动热点之间切换网络，只想访问少数目标网站 | 开启工具后目标网站可能可用，但其他原本可访问的网站异常；停止工具后仍不确定系统网络是否恢复 | 可以放心启停 GoGuo；失败时知道发生了什么，不需要理解代理细节 |

## 3. Why now / 当前价值判断

`OPP-002` 是安全前置门槛。战略发现已确认当前阶段顺序为 `OPP-002 → OPP-004 → OPP-001`，原因是：

- `OPP-001` 的目标站点自动可达能力会触发系统网络状态变更；若没有 baseline/restore，自动化越强，误伤风险越大。
- `OPP-004` 涉及 Windows 与 WSL/Linux 协同环境差异；若 baseline 没有覆盖 WSL 侧可见状态，后续诊断可能只证明 Windows 浏览器可用，不能证明开发工作流可用。
- 现有原型已具备 MonitorServer、DecisionEngine、ProxyCoreManager、StateManager 等可复用基础，但缺少产品化的安装后评估、Proxy Guard 和恢复闭环。
- 用户对“自动修复”天然有焦虑。能停、能恢复、能解释，是用户允许 GoGuo 接管网络状态的前提。

因此，本轮 discovery 不追求更广泛协议支持、完整 UI 或站点级规则，而是先证明 GoGuo 可以成为一个可停用、可恢复、可解释的本地网络可达性工具。

## 4. 当前轮 wedge / 最小机会点

当前轮唯一主 wedge：

> 安装后网络评估与基线恢复：GoGuo 首次运行后先评估 Windows 与 WSL/Linux 协同环境的关键网络状态，先保存“安装前初始状态”作为只读参考快照，再通过向导协助用户完成必要的低风险网络调整并确认一个可用 baseline；后续停止服务、代理异常、自动恢复失败或网络切换造成状态偏离时，能够恢复到该 baseline，或明确提示无法自动恢复的项目与建议动作。

本轮将“初始状态”和“baseline”分开处理：初始状态是安装前只读快照，用于诊断和必要时提示人工恢复；baseline 是经过 GoGuo 评估、必要调整和用户确认后的可用状态，是后续自动恢复的主要目标。默认不承诺从 baseline 自动回退到初始状态，因为初始状态可能本身不可用或不完整；若用户明确要求回退，GoGuo 应输出差异清单和安全提示，而不是静默改回。

当前轮不把“自动让目标站点持续可用”作为完成标准。它只把自动可达能力所需的安全底座做成可进入规格的产品问题。

## 6. 关键假设与风险

### Desirability

| 假设 | 风险 | 本轮处理 |
|------|------|----------|
| 普通办公用户愿意在首次使用时接受一次网络评估与 baseline 确认。 | 若首次流程过重，用户可能直接放弃。 | probe 中验证“最小解释 + 一次确认”的接受度。 |
| 开发者愿意把 Windows 与 WSL/Linux 协同环境的网络差异交给 GoGuo 诊断。 | 若用户认为工具越权或不可控，会拒绝接入。 | 明确只读评估优先，自动写入需单独确认。 |

### Feasibility

| 假设 | 风险 | 本轮处理 |
|------|------|----------|
| Windows 关键状态可可靠采集、保存、对比和恢复。 | 注册表、hosts、进程、TUN、DNS 缓存、防火墙等状态并非都能无损恢复。 | 将状态项分为可自动恢复、可检测不可恢复、暂不覆盖三类。 |
| WSL/Linux 关键状态可低风险评估。 | WSL 发行版、shell 配置、环境变量、Git 配置和 VS Code Remote 状态差异大。 | 当前只承诺评估和差异提示；自动配置接管留给 `OPP-004`。 |
| Proxy Guard 能发现工具链异常并触发恢复路径。 | 只检查进程存活可能漏掉客户端异常、监控服务异常、端口死锁、系统代理残留或代理核心健康 API 不可用。 | probe 需要覆盖 GoGuo 客户端、监控服务、代理核心、相关子进程、端口、系统代理、健康 API 的组合判断。 |

### Usability

| 假设 | 风险 | 本轮处理 |
|------|------|----------|
| 用户能理解“恢复到已确认 baseline”而不是“恢复到出厂原状”。 | 文案不清会导致用户对恢复能力过度期待。 | discovery 到 spec 的 Bridge 中要求明确 baseline 解释与确认文案。 |
| 失败解释能降低焦虑。 | 如果提示只输出技术日志，普通用户仍无法处理。 | 失败提示应包含原因、已尝试动作、下一步建议和是否需要手动处理。 |

### Viability / 合规与信任

| 假设 | 风险 | 本轮处理 |
|------|------|----------|
| 本地审计能同时满足诊断和隐私边界。 | 审计过细可能记录访问明细或敏感网络信息。 | 审计只记录本地系统改动、恢复动作、错误码和时间，不记录访问内容明细。 |
| 把 GoGuo 定位为本地网络可达性诊断与恢复工具，可以降低自动接管网络的信任门槛。 | 若公开表达仍承诺“自动翻墙”，会放大合规和信任风险。 | 本轮文档不采用违规导向表述，保持本地可达性、恢复、诊断叙事。 |

## 7. 候选方向与排除项

### 当前轮候选方向

| 方向 | 说明 | 当前判断 |
|------|------|----------|
| S1：安装后 baseline 评估与用户确认 | 首次运行后采集安装前初始状态快照，执行基础可达性检查，向导式协助用户完成必要调整，并形成用户确认 baseline。 | 主路径，必须进入 Bridge to Spec。 |
| S2：baseline 偏离检测与恢复 | GoGuo 停止、代理异常、网络切换或恢复失败时，对比当前状态和 baseline，自动恢复可恢复项。 | 主路径，必须进入 Bridge to Spec。 |
| S3：Proxy Guard | 守护 GoGuo 客户端、监控服务、代理核心、相关子进程、端口、系统代理与健康 API 一致性，避免系统代理指向死服务。 | 主路径，必须进入 Bridge to Spec。 |
| S4：最小本地审计与失败解释 | 记录系统改动和恢复结果，失败时展示原因、尝试次数、建议动作。 | 主路径，必须进入 Bridge to Spec。 |
| S5：WSL/Linux 与 Windows 协同环境只读评估 | 识别 Windows、WSL/Linux 单独部署或同机协同部署时的代理感知状态差异，先输出诊断提示。 | 作为与 `OPP-004` 的接口边界进入 Bridge，不在本轮承诺自动接管。 |

S5 需要覆盖三类常见部署组合：仅 Windows 部署并期望 WSL 侧可诊断；仅 WSL/Linux 部署并期望 Windows 侧至少可提示；Windows 与 WSL 同机协同部署并期望两侧状态一致可见。本轮只做只读评估、差异识别和建议输出，不做跨环境自动写入。

### 当前轮排除项

| 排除项 | 剪枝理由 |

| 目标站点规则自动生成与 P95 恢复 <= 10s | 属于 `OPP-001`；本轮只要求恢复安全底座。 |
| 完整 Web/桌面 UI | 属于 `OPP-003`；本轮只要求足以确认 baseline、展示恢复结果和失败解释的最小交互语义。 |

| 记录完整访问日志 | 与隐私和合规边界冲突；本轮只记录系统改动审计和诊断摘要。 |

## 8. 建议 probe / 验证优先级

| 优先级 | Probe | 验证对象 | 成功信号 |
|--------|-------|----------|----------|
| P0 | Baseline 状态项清单与读写矩阵 | Windows hosts、系统代理、代理核心、GoGuo 客户端、监控服务、相关子进程、端口、TUN、DNS 缓存、WSL/Linux 代理感知状态 | 每个状态项被分类为“可采集/可恢复/可检测不可恢复/暂不覆盖”，且有理由。 |
| P0 | 停止服务恢复演练 | 从 GoGuo 活跃状态停止到 baseline | 可恢复项回到 baseline；不可恢复项有明确提示；非目标直连站点不低于 baseline。 |
| P0 | Proxy Guard 异常注入 | mihomo 崩溃、GoGuo 客户端异常、监控服务异常、相关子进程残留、端口不可用、系统代理残留、健康 API 不通 | 能检测异常并触发恢复、续跑或提示，不让系统代理长期指向死服务。 |
| P0 | 本地审计样例走查 | 审计内容与隐私边界 | 审计能说明“改了什么、何时改、是否恢复”，不记录用户访问明细。 |
| P1 | 首次评估交互走查 | 普通办公用户和开发者是否理解 baseline | 用户能解释“baseline 是我确认过的可用状态”，而不是误解为系统出厂状态。 |
| P1 | WSL/Linux 协同环境只读诊断演练 | Windows 与 WSL/Linux 网络状态差异 | 至少能识别浏览器可用但 WSL/CLI 不可用的差异，并输出下一步建议。 |
| P1 | 模拟用户访谈 | 普通办公用户、PC 端开发者、进阶用户、运维调试用户对 baseline、初始状态、失败解释的理解 | 3-5 个角色样本能复述 baseline 含义；关键假设形成初始置信度。 |

这些 probe 的正式实验记录可在后续 `hf-experiment` 或规格验证设计中落盘。本 discovery 只定义验证方向和优先级。

## 9. 成功度量

- Desired Outcome：用户可以放心启停 GoGuo；停止、代理异常或自动恢复失败后，Windows 与 WSL/Linux 协同环境的关键网络状态回到用户确认的可用 baseline，或明确知道哪些项无法自动恢复以及下一步动作。
- North Star 锚定：项目当前无正式 North Star 声明；本轮锚定战略发现 Objective 1 “验证 GoGuo 的 PC 端基线恢复与可达性闭环”。
- Leading 指标：
  - baseline 状态项分类覆盖率。
  - 停止服务后可恢复项恢复成功率。
  - Proxy Guard 异常检测命中率。
  - 重启后未完成恢复动作续跑成功率。
  - 模拟用户访谈中关键假设置信度变化。
  - 失败提示中原因、已尝试动作、建议动作三要素完整率。
- Lagging 指标：
  - 用户停止 GoGuo 后非目标直连站点成功率不低于 baseline。
  - 用户在恢复失败后能根据提示完成下一步处理的比例。
- Success Threshold：
  - Windows 侧 hosts、系统代理、代理核心、GoGuo 客户端、监控服务、相关子进程、端口、DNS 缓存、TUN 状态至少被分类并纳入 baseline 差异检查；无法自动恢复的项目必须明确提示。
  - WSL/Linux 侧至少完成只读评估边界定义，能识别 WSL 代理感知状态与 Windows 状态差异；自动写入不作为本轮成功条件。
  - 停止服务与 Proxy Guard 异常两类场景中，可恢复项 100% 回到 baseline；不可恢复项 100% 给出原因与建议动作。
  - 工具重启后，必须先检查并继续处理上次停止或异常前未完成的恢复动作，再进入后续自动操作。
  - 本地审计只记录系统改动和恢复动作，不记录访问内容明细。
- Non-goal Metrics：
  - 不追求目标站点 P95 恢复 <= 10s。
  - 不追求新增协议数量。
  - 不追求完整 GUI 覆盖率。
  - 不追求移动端、多设备同步或自建服务稳定性。

### Jobs Stories

- When 我安装并首次运行 GoGuo，准备让它自动处理目标网站访问时，I want 先确认它理解我当前电脑和开发环境的可用网络状态，so I can 之后随时停用它而不担心电脑网络被改坏。
- When 自动恢复失败或代理核心异常时，I want GoGuo 告诉我它改了什么、恢复了什么、还剩什么需要我处理，so I can 不必在 hosts、系统代理、WSL、DNS 和进程之间盲目排查。
- When 我在 Windows 浏览器和 WSL/CLI 中看到不同访问结果时，I want GoGuo 至少能指出 Windows 与 WSL 的网络状态差异，so I can 知道问题是工具未覆盖、配置冲突，还是网络本身不可用。

| Push of the situation | 现有工具停止后仍可能影响正常访问，用户对自动接管网络缺乏信任。 |
| Pull of the new solution | 安装后确认 baseline，后续所有自动修复都能回到这个可解释状态。 |
| Anxiety of the new solution | 用户担心工具修改系统代理、hosts、TUN 或 WSL 配置后无法复原。 |
| Habit of the present | 用户继续手工切换代理、重启工具、清 hosts、重启电脑，虽然麻烦但感觉更可控。 |

## 11. OST Snapshot

Desired Outcome：用户可以放心启停 GoGuo；停止、代理异常或自动恢复失败后，关键网络状态回到用户确认的可用 baseline，或明确知道无法自动恢复项和下一步动作。

Opportunity A：首次运行时，用户不知道 GoGuo 会基于什么状态恢复网络。

- Solution A1：安装后网络评估与 baseline 确认。
  - Assumption：用户愿意接受一次简短评估和确认。
  - Probe：用静态流程或命令行原型让 3-5 个目标用户解释 baseline 含义。
- Solution A2：baseline 状态项分层说明。
  - Assumption：把状态项分为可恢复、可检测不可恢复、暂不覆盖能降低用户误解。
  - Probe：审查状态项清单和用户文案是否能被非专家理解。

Opportunity B：工具停止或异常后，系统网络可能停留在偏离 baseline 的状态。
<!-- [?] id:05;status:close;date:2026-04-30T11:19  工具停止或异常后，建议：重启工具，应优先对之前停止|异常前未处理的工作，继续处理完后，再进行后续工作-->

- Solution B1：停止服务时执行 baseline 差异检查和恢复。
  - Assumption：关键 Windows 状态可被可靠采集和恢复。
  - Probe：执行停止服务恢复演练，覆盖 hosts、系统代理、进程、端口、DNS 缓存、TUN。
- Solution B2：Proxy Guard 异常检测。<!-- [TODO] id:06;status:close;date:2026-04-30T11:19  Solution B2中，除了mihomo服务的异常，还包括工具客户端、其他进程的异常-->
  - Assumption：GoGuo 客户端、监控服务、代理核心、相关子进程、端口、系统代理和健康 API 的组合足以发现主要失效场景。
  - Probe：注入 mihomo 崩溃、GoGuo 客户端异常、监控服务异常、子进程残留、端口不可用、系统代理残留等异常。
- Solution B3：恢复动作续跑。
  - Assumption：把未完成恢复动作持久化为最小恢复任务记录，能降低异常退出后的网络残留风险。
  - Probe：在恢复流程中强制终止工具，重启后检查是否先续跑未完成恢复动作，再允许后续操作。

- Solution C1：WSL/Linux 协同环境只读评估与差异提示。
  - Assumption：不自动写入也能为 `OPP-004` 提供足够产品价值和边界证据。
  - Probe：在 WSL shell、Git/GitHub、VS Code Remote/CLI、agent 订阅请求中做只读诊断演练。

## 12. Bridge to Spec

若本 discovery 通过 `hf-discovery-review`，建议进入 `hf-specify` 的范围如下。

### 推荐进入规格的范围边界

- 安装后网络评估流程。
- 用户确认 baseline 的最小交互语义。
- baseline 状态项清单、分类、保存、对比与恢复边界。
- 安装前初始状态只读快照，用作诊断和人工恢复参考，不作为默认自动恢复目标。
- 停止服务恢复流程。
- Proxy Guard 的产品行为边界：检测什么、何时恢复、何时续跑、何时提示。
- 工具重启后的未完成恢复动作续跑语义。
- 本地审计边界：记录系统改动和恢复动作，不记录访问内容明细。
- 失败解释最小语义：原因、已尝试动作、尝试次数、建议动作、是否需要手动处理。
- Windows 与 WSL/Linux 协同环境的只读评估边界，作为 `OPP-004` 的前置输入。
- 后续 probe 需要用 3-5 个目标用户角色或真实用户访谈样本，对关键假设形成初始置信度。

### 可直接转成规格输入的稳定结论
<!-- [?] id:09;status:close;date:2026-04-30T11:19  整体同意，但对linux/wsl单独处理有疑义-->
- GoGuo 恢复目标是“用户确认的可用 baseline”，不是未经确认的系统原状。
- 安装前初始状态应被保存为只读参考快照，用于差异解释和人工恢复提示；默认自动恢复目标仍是用户确认 baseline。
- 自动切换系统网络状态前，必须先有 baseline、恢复路径、二次确认和本地审计。
- 停止服务必须触发 baseline 差异检查；不能只停止进程。
- 工具启动时必须优先检查是否存在未完成恢复动作；若存在，应先续跑或提示处理，再进入后续自动操作。
- Proxy Guard 必须覆盖 GoGuo 客户端、监控服务、代理核心、相关子进程异常与系统代理残留风险。
- 审计不能记录用户访问内容明细。
- Windows 与 WSL/Linux 协同环境本轮先作为只读评估和差异提示，不承诺完整自动写入。

## 13. 开放问题

### 阻塞项

| 问题 | 为什么阻塞 | 建议处理 |
|------|------------|----------|
| baseline 状态项清单是否足够完整。 | 若状态项漏掉，恢复承诺会不可信。 | 在 `hf-specify` 前或规格早期明确状态项矩阵。 |

| 问题 | 说明 |
|------|------|
