结合控制论和微软Assert开源项目对本项目spec到代码实现的自动化测试思考

Feature 001-004 实现了，但代码与规格不符；又在用 F101-113 来修复——这个模式本质上是在用无限的“打补丁”来弥补最初的“理解偏差”。

要终止这个循环，需要在当前阶段先止血、再重建，而不是继续在第 101、102 个补丁上修修补补。

一、为什么你的项目会陷入“Feature 101–113 无穷循环”
首先需要确认一个核心洞察：代码之所以能运行却偏离规格，根源不是 AI 写错了代码，而是整个项目缺少“从意图到代码”的可追溯系统。

用本对话中此前讨论的三层框架来分析 goguo 的现状：

层级	现状	问题
Spec（意图/规格）	早期有设计规格，但Feature 001–004实现后，代码与规格已产生明显偏离，规格文档未同步更新，实际系统行为与原始设计文档的差距已在迭代中失控	意图层已经“失锚”——不知道当前代码对应的是哪一版意图
代码实现	Feature 001–004功能可用，但实现方式与原始规格不符（“跑偏”）	生成时缺少意图约束
反馈/测试	缺失一个可自动判定“功能是否正确实现”的测试体系，依赖人肉判断	无法规模化区分L2（逻辑错误）和L3（意图偏差）
Feature 101–113 的本质是：在代码层逐个打补丁，却从未判断过这些补丁是在修补语法/逻辑错误（L2），还是在修补“当初就理解错的需求”（L3）。

两者混在一起，所以“感觉没有尽头”。引入三层分级 + 意图驱动的测试框架，不是为了“测得更快”，而是为了区分“补丁”的类型，让L3问题——也就是当前循环的根源——在外环被一次性解决，而不是在内环无限消耗。

二、在当前阶段中途引入的核心原则
在 goguo 当前这种“代码已实现、补丁在持续”的项目状态下，有三条原则必须遵守：

不推翻全部代码：不需要从零开始重写 Feature 001–004，而是将当前代码看作 “当前意图的近似表达” ，用测试框架反推出“它实际在做什么”，再与“它应该做什么”对齐。

先建立意图基准线，再分类补丁：补丁 101–113 中含有大量宝贵的手动修正信号——不要丢弃，而是用它们来反哺 Spec。

L2 问题让 AI 自动修，L3 问题让系统改 Spec：这是终止“Feature 101-N”循环的关键——不是继续打补丁，而是让系统学会区分问题的层级。

三、四步分阶段落地路线图
第一阶段：止血——用“补丁反向萃取”重建 Spec（1–3 天）
在所有新 Feature 之前，先做一次 “意图–代码对齐盘点” ，工具采用 ASSERT + Spec Extraction。

步骤：

拉取 Feature 001–004 的原始需求文档/设计规格，以及对应的最终代码。

用 ASSERT 的 systemize 阶段 处理规格文档，生成结构化的行为块（pattern blocks）——即使文档已经部分过时，ASSERT 也能从中提取出“原意图骨架”，不需要人工从零撰写。

将 Feature 101–113 的每个补丁输入 ASSERT，通过 inference 与 judge 阶段批量分析：

这个补丁修复的是什么问题？

——是语法/类型/编译层面的硬性错误？

——还是“功能行为与需求描述不符”？

生成一份 意图基线报告：每个 Feature 的“原始意图”和“当前实际行为”的差异对照表。

预期产出： 不写任何新代码，就把 001–004 + 101–113 的状态压缩成一份可量化的意图差距清单。有了这份清单，才知道该停在哪、该往哪改。

第二阶段：建立分级归因的测试层（3–7 天）
在项目原有的 CI/测试体系中，接入 ASSERT 的闭环。

具体配置示例：

python
# ⚙️ 安装
# pip install assert-ai==0.1.0

from assert_ai import AssertEval

# 1️⃣ Systematize：将 Feature 的需求描述转化成结构化的行为模式
#   👤 输入自然语言意图 → 生成可执行的测试规格
eval = AssertEval.from_spec("""
Feature 001: 用户登录时，输入正确凭证应跳转至仪表盘；
Feature 002: 密码错误三次应锁定账户15分钟；
Feature 003: 会话超时应自动登出；
Feature 004: …
（此处用第一阶段产出的意图基线报告填充）
""")

# 2️⃣ Test Set & Inference：对 goguo 的系统进行测试
results = eval.run(
    target=lambda: call_goguo_api(...),   # 👈 调用你的 goguo 代码
    model="openai/gpt-5.4",               # 或其他任何通过 LiteLLM 支持的模型
    num_test_cases=100                    # 生成 100 个测试用例/变体
)

# 3️⃣ Judge：得到分级评分报告
for test_case in results:
    print(f"✅ {test_case.passed} | ❌ 失败原因：{test_case.reason} | 🔍 场景：{test_case.scenario}")
ASSERT 的 test-set 阶段会自动生成正例（应允许的请求）、负例（应触发策略边界的请求）和边界用例，输出结构化的测试报告，直接回答：哪些行为符合意图，哪些不符合。

与 CI 集成： 将 ASSERT 接入 CI 流程，每次代码变更自动触发测试，实时反馈“行为 vs. 意图”的匹配度，终止“提交后才发现问题”的被动修复模式。

第三阶段：引入 L2/L3 分级归因（关键——这是终止无限补丁的核心）
在上一阶段的 AssertEval 输出基础上，加入一个轻量级的分级判断逻辑，将测试失败归类为 L2（逻辑错误） 或 L3（意图偏差） 。

实现思路：

L2 判定规则：

单个测试用例失败，且失败原因可归因于具体代码片段（如返回了错误的值、条件分支不完整）；

相同模式下，单元测试覆盖范围内的局部问题。

L3 判定规则：

多个测试用例同时失败，且失败模式指向 需求描述本身的不一致性；

或：对同一个意图描述，ASSERT 生成的不同测试用例均持续失败，即使代码已经部分修正。

一旦识别为 L3 意图错误，系统 不进入代码补丁循环，而是：

将失败信息回传到外环（需求/设计层）

触发 设计文档/意图 IR 的重构

再重新进入代码生成

这是解决“Feature 101–113”循环的核心切换：不是用第 102 个补丁去修第 101 个补丁的逻辑偏差，而是追溯到意图层一次性修正。

第四阶段：建立“补丁优先级的自动排序”（可选但推荐，2–3 天）
Feature 101–113 的大量补丁不是要丢弃，而是可以反哺成知识。借助 ABTest 的思路——它通过从真实的失败报告中挖掘交互模式和 Action 类型，生成可复用的测试模板——可以：

将 Feature 101–113 的每个补丁描述，作为 user-reported anomalies 输入；

ABTest 自动从中提取 Interaction Patterns（如“反复失败的边界条件类型”）和 Action types（如“返回值校验缺失”）；

生成对应的 fuzzing 测试用例，使未来类似的问题被提前发现，而不是等到补丁 114 才处理。

五、一个重要的工程建议
在 goguo 目前的项目阶段，不建议直接推翻代码重写 Feature 001–004。更好的方式是：

将当前代码视为 “当前规格的可执行表达” ；

用 ASSERT 反推出“代码实际满足的规格”；

将这份“实际规格”与“原始设计规格”对比；

差异点 → 精确的 L3 意图修正项，而不是全局推倒重来。

这种做法可以保证现有代码的稳定性，同时将偏离的部分转化为可追溯的工程任务，而不是模糊的“Bug 列表”。

七、总结
goguo 项目目前的核心困境不是“代码质量不够好”，而是 “意图–代码–测试”三层之间缺乏闭环。Feature 101–113 的出现，是在用代码层的无限修正，去补偿意图层的缺失。

要终止这个循环，关键在于：让 L3（意图）问题显性化，并通过 ASSERT 等规约驱动测试框架，在意图层一次性解决，而不是让 AI 在内环用补丁反复试探。

当前项目阶段的核心动作是 “止血”而非“重写” ：用 ASSERT 对已有功能做一次规约对齐分析，明确每个 Feature 的实际行为与原始设计意图之间的偏差，将偏离项归为 L2（逻辑/实现）或 L3（意图/需求），L2 让 AI 在内环自动修正，L3 回退到设计层重构 Spec。只有切断 L3 问题在内环的反复消耗，才能走出 Feature 101–113 的无限循环。
