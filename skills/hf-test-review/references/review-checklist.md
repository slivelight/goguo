# Test Review Checklist

评审测试资产时，至少对以下 6 个维度逐项审查。每个维度内部评分 `0-10`，评分帮助区分轻微缺口与阻塞问题。

## 评分辅助规则

- 任一关键维度低于 `6/10` → 不得返回 `通过`
- 任一维度低于 `8/10` → 通常至少对应一条具体 finding

## 评审维度

| ID | 维度 | Pass Condition |
|---|---|---|
| `TT1` | fail-first 有效性 | `RED` 真的证明了行为缺口，`GREEN` 可回指到当前实现 |
| `TT2` | 行为 / 验收映射 | 测试覆盖关键行为，并能回指到任务或验收口径 |
| `TT3` | 风险覆盖 | bug-patterns 与关键边界、错误路径被显式覆盖 |
| `TT4` | 测试设计质量 | mock 只用于真正边界，测试独立、可重复、可读 |
| `TT5` | 新鲜证据完整性 | test review 基于当前会话可核实的 RED/GREEN evidence，而不是旧绿测 |
| `TT6` | 下游就绪度 | 测试质量足以支持 `hf-code-review` 的可信判断 |

### `TT1` fail-first 有效性

- `RED` 是否对应当前行为缺口？
- 是否真的在本次会话里看到预期失败？
- `GREEN` 是否来自本次实现，而不是原本就绿？

### `TT2` 行为 / 验收映射

- 测试是否覆盖任务关键行为？
- 是否能回指到任务 acceptance 或等价验收口径？
- 是否只验证了实现细节，而没有验证行为结果？

### `TT3` 风险覆盖

- bug-patterns 识别的风险是否被覆盖？
- 是否覆盖边界、错误路径、权限差异、空值或状态切换？
- 是否只有 happy path？

### `TT4` 测试设计质量

- mock 是否限定在真正边界？
- 测试是否独立、可重复、命名清晰？
- 是否出现“测试 mock 的行为”而不是“测试系统行为”？

### `TT5` 新鲜证据完整性

- 当前结论是否基于可核实的 RED/GREEN 证据？
- 是否在用旧输出或口头陈述替代当次结果？
- handoff 中的测试证据是否足够冷读？

### `TT6` 下游就绪度

- 当前测试质量是否足以让 `hf-code-review` 判断代码正确性？
- 是否仍缺少会直接污染 code review 的测试缺口？

## Anti-Pattern 检测

| ID | Anti-Pattern | 检测信号 | 正确做法 |
|---|---|---|---|
| `TA1` | born-green 测试 | 测试一跑就绿，没有证明行为缺口 | 先建立有效 RED，再谈 GREEN |
| `TA2` | happy-path-only | 只测正常路径，不测错误或边界 | 补齐失败、边界和权限场景 |
| `TA3` | mock overreach | 大量断言 mock 调用而非行为结果 | 仅在真实边界使用 mock |
| `TA4` | no acceptance link | 测试存在，但无法回指到任务 acceptance | 补 trace anchor 或验收映射 |
| `TA5` | stale evidence | 用旧绿测或口头结果替代当次输出 | 使用当前会话 fresh evidence |
