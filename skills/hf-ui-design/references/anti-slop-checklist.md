# 反 AI Slop 设计清单

UI 设计文档容易出现"看起来像 AI 默认产物"的视觉与内容惯性。本清单列出常见 slop 模式与应对方式，用于设计阶段自检与 `hf-ui-review` 的 anti-pattern 检测。

来源：综合行业一般经验与 Anthropic Claude `frontend-design` / `Claude-Design` 系统提示词中针对默认审美的反向规约（"avoid AI slop tropes", "every element should earn its place", "intentional direction, not generic defaults"）。

## 0. 总原则

- **每个元素必须 earn its place（自证存在价值）**：不能"看起来空就填东西"，留白本身是设计决策
- **缺资源时占位优于劣质仿制**：在没有真实图标/图片/数据/品牌资产时，用 placeholder（带说明的占位框、灰底文字标签、`{{ image:hero-product }}` 锚点）说明意图，比"用 SVG 自己画一个像样的"更诚实
- **不在规格之外擅自增加内容**：感觉某区域"应该再加点东西"时，先问，而不是先加
- **匹配既有视觉语汇**：本设计若是嵌入既有产品，必须先冷读既有 UI 的色板/字号/圆角/密度/动效/微文案语气，再决定是沿用还是显式偏离

## 1. 视觉 slop 反模式

| ID | 反模式 | 检测信号 | 替代做法 |
|---|---|---|---|
| `S1` | **渐变滥用** | 大面积彩色背景渐变、主按钮使用渐变、卡片用渐变描边 | 默认 solid 背景；只在品牌已确立渐变语言、或确有功能性表达时使用，且时长/角度/色域走 token |
| `S2` | **左边竖线 + 圆角容器** | 所有 callout / alert / quote 都是"左侧 4px 彩条 + rounded card" | 信息层级用排版（字重 / 行距 / 间距）和语义色背景区分；不依赖单一容器范式 |
| `S3` | **默认字体套用** | typography 直接是 Inter / Roboto / Arial / Helvetica / 系统栈，且无理由说明 | 显式说明字体选择理由；若用 system stack 也要说明（如离线场景、性能预算、多语支持范围）；避免"反正这些字体安全" |
| `S4` | **默认 AI 配色** | 紫色/紫蓝渐变主色、品牌 #6366F1 / Tailwind violet-500 直接当主色 | 从规格语境（行业、目标受众、情绪关键词）反推主色；扩展色板用 OKLCH 在既有色域里推导，避免随手调 RGB |
| `S5` | **装饰性 SVG / Emoji** | 用 SVG 自画"科技感"插画、用 emoji 当图标、状态用 ✅⚠️❌ | 缺图标 → placeholder 框 + 命名（如 `{{ icon:warning }}`）；emoji 仅在品牌明确使用 emoji 时用 |
| `S6` | **千篇一律的 dashboard** | 顶 nav + 左 rail + 卡片网格 + 趋势图 + KPI 大数字，无场景化裁剪 | 先按规格中的核心用户任务决定主视图（编辑器型 / 列表型 / 画布型 / 单焦点型 / 工作流型），dashboard 是默认值不是答案 |
| `S7` | **Glassmorphism / 高斯模糊滥用** | 模糊 + 半透明 + 边框光泽，无障碍对比度直接崩 | 默认不使用；若用，明确 `backdrop-filter` 在对比度计算中的处置与降级路径 |
| `S8` | **过度阴影 / 浮起卡片** | 同一页面 5+ 不同阴影深度，所有卡片都"浮起来" | shadow 走 token（最多 3 档），层级用 elevation 表达不是装饰 |

## 2. 内容 slop 反模式

| ID | 反模式 | 检测信号 | 替代做法 |
|---|---|---|---|
| `C1` | **填充式 hero copy** | "Empower your workflow", "Unlock seamless productivity", "Reimagine the way you ..." 类无信息文案 | 用规格中的真实价值主张；若不知道，写 `{{ copy:hero-headline-pending }}` 占位 |
| `C2` | **数据 slop** | 仪表盘塞满"+12.4%"、"2.3K active"等没有业务定义的数字、无意义图标徽标 | 每个数字/徽标必须可回指规格中的指标定义；不可回指就删 |
| `C3` | **多余 section 自动生成** | 在规格未要求时，自动加上"Testimonials / Features grid / FAQ / CTA banner / Footer 5 列" | 任何规格之外的 section 都需要先问用户；默认不加 |
| `C4` | **占位文本伪装成正文** | "Lorem ipsum"、"Sample card text" 直接出现在交付稿里 | 占位也要带语义，如 `{{ copy:product-description, ~80 chars }}`，让 reviewer 一眼分辨是占位还是定稿 |
| `C5` | **图标堆叠** | 每个 list item / 每个 menu 项都配一个图标，图标语义重复 | 图标只在能加速识别 / 表达状态 / 减少认知负担时用；不能"为对齐而图标" |

## 3. 结构 slop 反模式

| ID | 反模式 | 检测信号 | 替代做法 |
|---|---|---|---|
| `ST1` | **无来源的"Design System"** | 写了 token 表，但没说 token 出处（沿用 / 扩展 / 新增），也没说与既有产品的关系 | 每条 token 标注来源；新增 token 必须 ADR 记录理由 |
| `ST2` | **未声明的视觉宣言** | 选定方向只写"现代、简洁、有科技感"形容词，没落到 typography / color / spacing / motion 的具体决策 | 视觉方向 ADR 必须包含：定位句 + 4 类 token 的具体策略 + 不取舍另一方向的代价 |
| `ST3` | **状态切换无设计** | 状态矩阵列了 loading/empty/error，但没画状态间过渡（fade / skeleton 替换 / 内容布局变化）→ CLS 与节奏断裂 | 状态切换显式说明过渡方式与 reduced-motion 降级 |
| `ST4` | **happy-path-only IA** | 站点地图只画了授权用户主路径，没画未登录 / 无权限 / 空数据 / 错误页归属 | IA 中显式标注鉴权态、错误页、空态归属页面 |

## 4. 与既有 UI 协同时的反模式

仅在本设计是嵌入或扩展既有产品时适用：

| ID | 反模式 | 检测信号 | 替代做法 |
|---|---|---|---|
| `E1` | **未读既有视觉语汇就开始新设计** | 候选方向不引用任何既有产品的色板/字号/圆角/微文案语气 | 第一步就把既有 UI 的视觉 DNA 摘要写到文档里，再决定沿用 / 扩展 / 显式偏离（偏离需 ADR） |
| `E2` | **新增组件无既有对应物对照** | 新增 SearchInput / DataTable / Modal，但既有产品已有同类组件却未引用 | 在 Atomic 映射表中显式列出"既有产品对应组件 → 复用 / 扩展 / 替换 + 理由" |
| `E3` | **两套命名 / 两套 token 并存** | 新设计用了 `--brand-primary`，既有产品用 `--color-accent-500`，没说映射 | token 表第一列必须说明与既有体系的映射；不映射就显式声明这是分叉 |

## 5. 自检流程（写完 UI 设计稿后 5 分钟）

按顺序冷读：

1. **每段文案**：能否回指规格中的真实需求？不能 → 改成 `{{ copy:... }}` 占位
2. **每个 token**：能否说出出处（沿用 / 扩展 / 新增 + 理由）？不能 → 补 ADR 或下沉到既有 DS
3. **每个图标 / 插画 / emoji**：能否说出为什么是这个而不是没有？不能 → 改 placeholder 或删
4. **每个 section**：能否说出规格中哪条要求催生了它？不能 → 删或问用户
5. **整页冷读**：是否一眼像"通用 SaaS dashboard 模板"？是 → 回到规格找该产品的差异化锚点

## 6. 在 UI 设计文档中的体现

`ui-design.md` 的 `## 7. 视觉系统` 与 `## 9. Atomic 组件映射` 默认应能让 reviewer 冷读出本清单 1/2/3 节的合规性。文档无需复述本清单条款，但应：

- 在视觉方向 ADR 的"否决备选方案"段中，显式说明为什么没采用本清单 S1–S8 中的某些 slop 模式作为表达手段（若候选方向曾涉及）
- 在 `## 15. Microcopy 与语气`（若启用）中，明确占位文案与定稿文案的标记方式（如统一 `{{ copy:... }}`）
- 在文档顶部 `Peer` 块附近，若是嵌入既有产品，写一段"既有视觉语汇摘要"（色板 / 字号 / 圆角 / 密度 / 微文案语气），作为后续设计选型的对照基线
