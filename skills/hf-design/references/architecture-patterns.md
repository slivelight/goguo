# 架构模式选择指南

在了解技术上下文后、提出候选方案前，识别当前需求匹配哪些架构模式。这一步帮助缩小候选方案范围，避免凭直觉直接跳到一个方案。

## 模式维度判断

| 维度 | 判断依据 |
|------|---------|
| 项目规模 | Solo creator / 小团队 → 模块化单体优先；多团队 → 考虑微服务 |
| 数据特征 | 文档为主、文件系统可承载 → 无需数据库；复杂查询、事务性强 → 引入结构化存储 |
| 交互模式 | 请求-响应 → 分层架构；事件驱动 → 异步管道；混合 → 按模块分别选择 |
| 部署形态 | 本地 / CLI → 最简运行时；服务化 → 需要考虑启停、健康检查 |
| 成长预期 | 短期稳定 → 优先简单；快速演进 → 需要扩展点和模块隔离 |

## 常见模式对比

| 模式 | 适用场景 | 团队规模 | 核心权衡 |
|------|---------|---------|---------|
| 单体 | 简单领域 | 1-5 | 部署简单；局部难以独立扩展 |
| 模块化单体 | 复杂度增长中 | 1-10 | 模块边界清晰；仍是单一部署 |
| 微服务 | 复杂领域、大组织 | 20+ | 独立扩展；运维复杂度高 |
| 分层架构 | 请求-响应型系统 | 不限 | 职责分离；层次间可能泄漏 |
| 管道/事件驱动 | 异步处理流 | 不限 | 解耦好；调试和一致性难追踪 |
| 插件架构 | 可扩展工具平台 | 1-5 | 核心稳定、扩展灵活；插件隔离需要设计 |

## 面向本地单人维护项目的常见默认偏好

对 docs-first / workspace-first / agent-native / local-tooling 一类项目，通常优先：
- **模块化单体**（不引入微服务、消息队列、分布式数据库）
- **文件系统存储**（文档型数据、数据量 < 10 万条时不引入数据库）
- **最简运行时**（本地 / CLI 优先，不引入重型框架）
- **先简单，后升级**（明确触发升级条件：数据量、用户数、请求频率）

如果当前项目不是这类形态，不要机械沿用这组偏好；应回到上面的模式维度判断，根据团队规模、数据特征、部署形态和成长预期重新选择。

识别结果在设计文档的"架构模式选择"章节中说明：选了什么模式、为什么适合、它天然带来了什么限制。
