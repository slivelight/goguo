# 设计评审协作产品 discovery 草稿

- 状态: 草稿
- 主题: 设计评审协作

## 1. 问题陈述
独立设计师提交方案后，反馈分散在 IM、文档评论和评审会议里，导致反馈合并成本高、版本决策不透明。

## 2. 目标用户与使用情境
- 主要用户：独立设计师
- 协作角色：评审主管、跨职能 reviewer
- 核心情境：一次设计评审周期内，需要集中收集、整理与追踪反馈

## 3. Why now / 当前价值判断
当前团队的评审频率上升，反馈散落导致设计迭代延迟；如果不能先解决反馈收口，后续所有审批/追踪能力都会建立在脆弱流程上。

## 4. 当前轮 wedge / 最小机会点
第一轮只聚焦“统一反馈收口”：让同一轮评审的反馈集中进入一个可回读视图，而不是继续散落在多个渠道。

## 5. 已确认事实
- 设计师目前需要跨 3 个渠道手工汇总反馈
- 评审主管经常无法快速回答“这条反馈是否已处理”
- 当前轮不要求解决审批流、权限配置或报表

## 6. 关键假设与风险
- 假设：如果先统一反馈收口，评审效率会显著提升
- 假设：设计师愿意在一个统一入口查看与处理反馈
- 风险：若 reviewer 不愿切换入口，统一收口的采用率可能不足

## 7. 候选方向与排除项
- 候选方向 A：统一反馈收口视图
- 候选方向 B：自动提醒与催办
- 当前轮排除：配置化流程、BI 报表、审批编排

## 8. 建议 probe / 验证优先级
- 先验证设计师是否愿意在统一入口处理反馈
- 再验证主管是否能更快判断反馈处理状态

## 9. Bridge to Spec
### 可直接进入 formal spec 的稳定结论
- 第一轮只做统一反馈收口，不做审批流或报表
- 目标用户是独立设计师与评审主管
- 成功标准应围绕“反馈汇总与状态可回读”展开

### 仍保留为 assumptions 的内容
- 统一入口是否能提升采用率
- reviewer 是否接受新的反馈入口

### 当前不进入 spec 的 later ideas
- 自动催办
- 配置化评审流程
- BI 报表

## 10. 开放问题（阻塞 / 非阻塞）
- 非阻塞：后续是否需要把 Slack 评论同步进统一入口
