## 1. 概述

### 1.1 目的

本文档定义 GoGuo 目标站点规则配置与可达性诊断的需求规格。基于 Feature 001 的 baseline/restore 安全底座和 Feature 002 的 WSL/Linux 配置能力，实现"只处理目标站点，不破坏非目标站点"的核心产品价值。

### 1.2 范围

1. 目标站点列表管理
2. 基于站点列表自动生成 mihomo 规则
3. 默认直连策略（DIRECT 兜底 + 目标 PROXY）
4. 站点可达性持续诊断
5. 不可达时的恢复动作或告警
6. 规则预览与覆盖
7. Windows 与 WSL/Linux 两侧规则同步

### 1.3 术语

| 术语 | 定义 |
|------|------|
| 目标站点 | 用户指定需要通过 GoGuo 确保可达的网站 |
| 站点规则 | 针对特定目标站点生成的 mihomo 分流规则 |
| 默认直连策略 | mihomo 配置中最终规则为 DIRECT，确保未匹配规则的流量直连 |
| 可达性诊断 | 定期探测目标站点是否可访问，并记录结果 |
| 站点定义 | 一个逻辑站点的完整描述，包含标识（id）、显示名称、分类域名列表（core/api/cdn/services 等）及可选的健康检查配置 |

### 1.4 目标用户

| 用户 | 本规格关注点 |
|------|-------------|
| 普通办公用户 | 预设模板快速配置、无需理解规则类型 |
| PC 端开发者 | 目标开发站点自动可达、内部站点不受影响 |
| 进阶用户 | 规则预览和覆盖能力 |

### 1.5 成功标准

| # | 标准 | 验证方式 |
|---|------|----------|
| SC-1 | 目标站点在网络切换后 P95 恢复 <= 10s；需切换代理节点时 <= 30s | 网络切换场景计时验证 |
| SC-2 | 非目标直连站点可达性不低于 baseline | 规则生效前后非目标站点对比 |
| SC-3 | 默认 DIRECT + 目标 PROXY 策略下非目标站点不经过代理 | 流量路径检查 |
| SC-4 | Windows 与 WSL/Linux 两侧目标站点均可访问 | 两侧分别验证目标站点可达性 |
| SC-5 | 目标站点不可达时提供五要素诊断提示 | 注入不可达场景，检查提示 |

## 2. 功能需求

### 2.1 目标站点列表管理

#### FR-2.1.1 站点添加与删除

**要求**:
- FR-2.1.1-R1: 用户可通过站点标识（如 `github`）或域名（如 `github.com`）添加目标站点
- FR-2.1.1-R2: 用户可删除已添加的目标站点
- FR-2.1.1-R3: 添加时若匹配已知站点定义，应按站点定义展开所有关联域名；若为独立域名，必须自动识别域名类型（精确域名 / 通配域名），并生成对应规则类型（DOMAIN / DOMAIN-SUFFIX）
- FR-2.1.1-R4: 站点列表变更必须触发规则重新生成
- FR-2.1.1-R5: 站点列表变更必须记入审计
- FR-2.1.1-R6: 对于不在已知站点定义中的域名，应尝试自动检测相关域名并向用户建议（参照站点定义结构），经用户确认后可作为自定义站点定义保存

### 2.2 规则自动生成

#### FR-2.2.1 规则生成逻辑

**要求**:
- FR-2.2.1-R1: 基于目标站点列表，自动生成 mihomo 规则配置
- FR-2.2.1-R2: 每个目标站点按其站点定义展开的域名列表生成 DOMAIN 或 DOMAIN-SUFFIX 规则，策略为 PROXY
- FR-2.2.1-R3: 规则末尾必须是 MATCH,DIRECT（默认直连兜底）
- FR-2.2.1-R4: 生成的规则配置必须通过 mihomo 配置语法校验后才能应用
- FR-2.2.1-R5: 规则生成失败时不得应用，必须保留当前规则并提示用户

#### FR-2.2.2 默认直连策略保障

**要求**:
- FR-2.2.2-R1: mihomo 配置中，目标站点规则之后的最终规则必须是 DIRECT
- FR-2.2.2-R2: 应用规则后，必须立即验证非目标直连站点可达性不低于 baseline
- FR-2.2.2-R3: 若非目标站点可达性低于 baseline，必须自动回退到上一份规则配置，并通过用户可感知的方式通知（如系统通知、状态栏变更提示），告知本次配置未生效及回退原因

### 2.3 站点可达性诊断

#### FR-2.3.1 持续诊断

**要求**:
- FR-2.3.1-R1: 必须定期探测每个目标站点的可达性
- FR-2.3.1-R2: 探测间隔应可配置，默认值在设计阶段确定
- FR-2.3.1-R3: 探测结果必须记录：时间戳、目标站点、可达/不可达、响应时间
- FR-2.3.1-R4: 探测失败不得产生用户可见的副作用（如弹窗、日志刷屏）

#### FR-2.3.2 不可达恢复

**要求**:
- FR-2.3.2-R1: 目标站点不可达时，应先尝试通过切换代理节点恢复
- FR-2.3.2-R2: 切换节点后仍不可达时，必须提供五要素诊断提示（原因、已尝试动作、尝试次数、建议动作、是否需要手动处理）
- FR-2.3.2-R3: 恢复动作必须记入审计
- FR-2.3.2-R4: 目标站点持续不可达超过阈值时，应降低探测频率以避免资源浪费
- FR-2.3.2-R5: 必须维护可切换的代理节点池，支持多个备选节点
- FR-2.3.2-R6: 代理节点池必须定期执行健康检查，验证节点可用性
- FR-2.3.2-R7: 连续健康检查失败的节点必须从可用池中移除（退出机制），移除动作记入审计

### 2.4 规则预览与覆盖

#### FR-2.4.1 规则预览

**要求**:
- FR-2.4.1-R1: 规则应用前必须提供预览，展示即将生效的规则列表
- FR-2.4.1-R2: 预览中应标注每条规则对应的站点名称和策略（PROXY/DIRECT）

#### FR-2.4.2 规则覆盖

**要求**:
- FR-2.4.2-R1: 进阶用户可对自动生成的规则进行覆盖（如将某目标站点从 DOMAIN 改为 DOMAIN-SUFFIX，或添加 IP-CIDR 规则）
- FR-2.4.2-R2: 覆盖的规则在规则重新生成时必须保留（不被自动规则覆盖）
- FR-2.4.2-R3: 覆盖操作必须记入审计

### 2.5 Windows 与 WSL/Linux 两侧规则同步

#### FR-2.5.1 规则生效

**要求**:
- FR-2.5.1-R1: 规则变更必须在 Windows 和 WSL/Linux 两侧同步生效
- FR-2.5.1-R2: 在"协同部署"下，两侧必须使用相同的规则配置
- FR-2.5.1-R3: 在"仅 Windows"或"仅 WSL/Linux"部署下，仅在对应侧生效
- FR-2.5.1-R4: 两侧规则生效后，必须分别验证目标站点可达性

### 2.6 规则变更与 baseline 安全

#### FR-2.6.1 规则变更安全

**要求**:
- FR-2.6.1-R1: 规则变更（应用新规则）属于系统网络配置修改，必须遵循 Feature 001 的二次确认机制
- FR-2.6.1-R2: 规则变更前后必须执行 baseline 差异检查
- FR-2.6.1-R3: 停止服务时，必须在 baseline 恢复流程中移除目标站点规则，恢复到 baseline 代理配置

|----|------|------|
| CON-1 | 规则策略为"默认 DIRECT + 目标 PROXY"，不采用全局代理模式 | 战略发现确认的致命问题解决方案 |
