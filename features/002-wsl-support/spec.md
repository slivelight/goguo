## 1. 概述

### 1.1 目的

本文档定义 GoGuo PC 端 Linux/WSL 支持的需求规格。基于 Feature 001 已完成的 baseline 评估与恢复基础，将 WSL/Linux 侧 4 个可恢复项从"只读评估"升级为"自动配置和恢复"，使 PC 端开发者无论从 Windows 还是 WSL 入口操作，都能获得一致的目标站点可达性。

### 1.2 范围

本规格覆盖以下产品行为：

1. WSL/Linux 侧 4 个可恢复项的自动配置与恢复
2. 三种部署组合的统一管理逻辑
3. WSL2 网络模式感知策略
4. WSL/Linux 侧二次确认和审计
5. WSL/Linux 侧失败解释
6. 与 Feature 001 的交接消费

### 1.3 术语

沿用 Feature 001 术语，补充：

| 术语 | 定义 |
|------|------|
| WSL2 NAT 模式 | WSL2 默认网络模式，WSL 运行在独立虚拟网络中，需显式配置代理 |
| WSL2 镜像模式 | Windows 11 22H2+ 可选模式，WSL 共享 Windows 网络栈，代理可能自动生效 |
| 协同部署 | GoGuo 运行在 Windows 或 WSL 其中之一，期望两侧均可访问目标网站 |

### 1.4 目标用户

| 用户 | 本规格关注点 |
|------|-------------|
| PC 端开发者 | WSL 中 git、npm、pip、VS Code Remote、agent 工具的网络可达性 |
| PC 端知识工作者 | WSL 中技术文档、协作工具的访问 |

### 1.5 成功标准

| # | 标准 | 验证方式 |
|---|------|----------|
| SC-1 | WSL/Linux 侧 4 个可恢复项 100% 可配置到 baseline 值并恢复 | 配置后逐项验证；恢复后逐项对比 |
| SC-2 | 协同部署下配置完成后 Windows 与 WSL 代理状态一致 | 两侧分别检测代理设置并对比 |
| SC-3 | WSL2 网络模式正确识别并根据模式选择适当配置策略 | 在 NAT 和镜像模式下分别验证行为 |
| SC-4 | WSL/Linux 侧自动配置失败时提供五要素失败提示 | 注入配置失败场景，检查提示内容 |
| SC-5 | 所有 WSL/Linux 侧操作记入统一审计流（与 Feature 001 共享） | 审查审计记录包含 WSL/Linux 侧操作 |

## 2. 功能需求

### 2.1 WSL/Linux 可恢复项自动配置

#### FR-2.1.1 配置范围

GoGuo 必须对 WSL/Linux 侧以下 4 个可恢复项执行自动配置：

| 状态项 | 配置方式 | 配置时机 |
|--------|----------|----------|
| WSL 代理环境变量 | 通过当前会话 `export` 使已运行进程立即感知代理设置（即时生效、会话级） | baseline 形成时用户确认调整；停止/异常恢复时 |
| WSL Git 代理配置 | 设置 `http.proxy`、`https.proxy` 到全局 Git 配置 | 同上 |
| WSL `/etc/resolv.conf` | 写入 DNS 服务器地址 | 同上 |
| WSL `/etc/environment` | 写入系统级代理环境变量（持久化、新会话自动生效） | 同上 |

> **注**: WSL 代理环境变量和 `/etc/environment` 是两个独立动作：前者通过会话 export 让已运行进程立即感知代理，后者通过文件写入让新启动的进程自动继承代理设置。两者配合确保当前会话和未来会话均可用。

**要求**:
- FR-2.1.1-R1: 配置操作必须仅在 Feature 001 分类为"可恢复项"的状态项上执行
- FR-2.1.1-R2: 配置前必须先读取当前值并保存到 baseline（若尚未保存）
- FR-2.1.1-R3: 配置操作必须通过 Feature 001 定义的二次确认机制
- FR-2.1.1-R4: 配置操作必须记入 Feature 001 定义的统一审计流

#### FR-2.1.2 配置执行

**要求**:
- FR-2.1.2-R1: 配置执行时必须逐项执行，每项执行后立即验证
- FR-2.1.2-R2: 单项配置失败不得阻塞后续项的配置尝试
- FR-2.1.2-R3: 配置结果（成功/失败/跳过 + 具体值）必须向用户展示
- FR-2.1.2-R4: 全部配置完成后，必须执行端到端验证：通过 WSL 侧 `curl` 或等价方式测试目标站点可达性

#### FR-2.1.3 配置恢复

**要求**:
- FR-2.1.3-R1: 停止服务或异常恢复时，必须将 4 个可恢复项恢复到 baseline 值
- FR-2.1.3-R2: 恢复后必须逐项验证当前值与 baseline 一致
- FR-2.1.3-R3: 恢复验证失败时不得静默跳过，必须记录并按 Feature 001 的失败解释机制提示用户

### 2.2 部署组合管理

#### FR-2.2.1 部署组合识别与适配

**要求**:
- FR-2.2.1-R1: 必须复用 Feature 001 FR-2.9.1 定义的部署组合识别逻辑
- FR-2.2.1-R2: 在"仅 Windows"部署下，不执行 WSL/Linux 侧配置操作
- FR-2.2.1-R3: 在"仅 WSL/Linux"部署下，仅管理 WSL/Linux 侧状态项，不依赖 Windows 侧能力
- FR-2.2.1-R4: 在"协同部署"下，同时管理 Windows 和 WSL/Linux 两侧状态项，并确保两侧一致
- FR-2.2.1-R5: 部署组合变更（如 WSL 被安装或卸载）时，必须重新识别并更新管理范围

#### FR-2.2.2 状态项管理复用

**要求**:
- FR-2.2.2-R1: WSL/Linux 侧状态项的采集、保存、对比和恢复逻辑必须复用 Feature 001 定义的核心抽象
- FR-2.2.2-R2: 新增 WSL/Linux 侧状态项不需要修改恢复流程核心逻辑（遵循 NFR-3.4-1 可扩展性）

### 2.3 WSL2 网络模式感知策略

#### FR-2.3.1 模式识别

**要求**:
- FR-2.3.1-R1: 必须检测当前 WSL2 网络模式（NAT / 镜像 / 未安装 WSL）
- FR-2.3.1-R2: 检测结果必须记入 baseline 快照

#### FR-2.3.2 策略选择

**要求**:
- FR-2.3.2-R1: **NAT 模式**：WSL 运行在独立虚拟网络中，必须显式配置代理环境变量、Git 代理和 DNS 设置
- FR-2.3.2-R2: **镜像模式**：WSL 共享 Windows 网络栈，代理设置可能自动生效。GoGuo 必须先验证 WSL 侧可达性：
  - 若可达性已满足 → 跳过代理环境变量和 Git 代理配置，仅记录当前状态到 baseline
  - 若可达性不满足 → 降级为与 NAT 模式相同的显式配置策略
- FR-2.3.2-R3: 选择的策略和理由必须记入审计

### 2.4 与 Feature 001 的交接消费

#### FR-2.4.1 只读评估产出消费

**要求**:
- FR-2.4.1-R1: 必须消费 Feature 001 FR-2.9 产出的状态项清单和分类结果
- FR-2.4.1-R2: 必须消费 Feature 001 FR-2.9 产出的差异报告，作为配置输入
- FR-2.4.1-R3: 若 Feature 001 的只读评估产出不存在（如升级安装、产出损坏），必须触发重新采集

#### FR-2.4.2 baseline 和审计复用

**要求**:
- FR-2.4.2-R1: WSL/Linux 侧配置值必须追加到 Feature 001 的 baseline 中，形成统一 baseline
- FR-2.4.2-R2: WSL/Linux 侧操作必须写入 Feature 001 的统一审计流，使用相同的审计格式
- FR-2.4.2-R3: WSL/Linux 侧恢复任务记录必须与 Feature 001 共享恢复任务持久化存储，确保续跑逻辑统一

### 2.5 失败解释

#### FR-2.5.1 失败提示

**要求**:
- FR-2.5.1-R1: WSL/Linux 侧配置或恢复失败时，必须使用与 Feature 001 一致的五要素失败提示（原因、已尝试动作、尝试次数、建议动作、是否需要手动处理）
- FR-2.5.1-R2: 建议动作必须针对 WSL 场景可执行（如"请在 WSL 中手动执行 `export http_proxy=...`"而非"配置代理"）
- FR-2.5.1-R3: 若失败原因为 WSL 发行版不支持或权限不足，建议动作应包含发行版特定的修复指导

## 3. 非功能需求

### 3.1 性能

| ID | 需求 | 验证方式 |
|----|------|----------|
| NFR-3.1-1 | WSL/Linux 侧 4 个可恢复项配置应在 10 秒内完成 | 计时验证 |
| NFR-3.1-2 | WSL/Linux 侧配置恢复应在 15 秒内完成 | 计时验证 |
| NFR-3.1-3 | WSL2 网络模式检测应在 3 秒内完成 | 计时验证 |

### 3.2 可靠性

| ID | 需求 | 验证方式 |
|----|------|----------|
| NFR-3.2-1 | WSL 配置写入失败不得影响 Windows 侧 baseline 和恢复能力 | WSL 不可用时触发停止恢复 |
| NFR-3.2-2 | WSL 发行版不支持时不得崩溃，必须降级为只读评估并提示 | 在非 Ubuntu/Debian 发行版上运行 |

### 3.3 安全

| ID | 需求 | 验证方式 |
|----|------|----------|
| NFR-3.3-1 | WSL/Linux 侧配置操作遵循与 Feature 001 一致的二次确认和审计机制 | 操作验证 |
| NFR-3.3-2 | 写入 /etc/resolv.conf 和 /etc/environment 需要适当权限，不得要求不必要的权限提升 | 权限检查 |

### 3.4 兼容性

| ID | 需求 | 验证方式 |
|----|------|----------|
| NFR-3.4-1 | 必须支持 Ubuntu（默认 WSL 发行版）和 Debian | 两个发行版上运行完整配置/恢复流程 |
| NFR-3.4-2 | 其他发行版（Arch、Fedora 等）以最佳努力支持，失败时降级为只读评估 | 非 Ubuntu/Debian 上运行并检查降级行为 |

### 3.5 可维护性

| ID | 需求 | 验证方式 |
|----|------|----------|
| NFR-3.5-1 | WSL/Linux 侧状态项管理代码与 Windows 侧共享核心抽象，新增平台时只需实现平台适配层 | 代码审查 |

## 4. 约束

| ID | 约束 | 原因 |
|----|------|------|
| CON-1 | 仅处理 Feature 001 已分类为"可恢复项"的 4 个状态项 | 已分类的状态项具备稳定读写基础 |
| CON-2 | 沿用 Feature 001 的 CON-1（平台无关：禁止自动网络配置接管，baseline 调整和恢复不在约束内） | 一致性 |
| CON-3 | 不修改 shell 配置文件（.bashrc/.zshrc/.profile） | 可检测不可恢复项，修改风险高 |
| CON-4 | 不配置包管理器代理（apt/pip/npm）和 Docker 代理 | 场景碎片化，4 个可恢复项已覆盖主流工具链 |
| CON-5 | 数据存储沿用 Feature 001 CON-5（统一安装根目录） | 一致性 |

## 5. 不在范围内

| 排除项 | 原因 |
|--------|------|
| shell 配置代理自动修改 | 可检测不可恢复项，影响用户自定义配置 |
| 包管理器代理配置 | 主流工具链通过环境变量已覆盖 |
| Docker 代理配置 | 特定用户群 |
| 独立 Linux 服务器（无 WSL）部署 | 远期假设 |
| 完整 Linux 桌面 GUI | OPP-003 |
