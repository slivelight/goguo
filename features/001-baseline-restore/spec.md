## 1. 概述

### 1.1 目的

本文档定义 GoGuo 安装后网络评估与基线恢复功能的需求规格。该功能是 GoGuo 自动网络切换能力的安全前置门槛——在任何自动修改系统网络状态的功能上线前，必须先证明 GoGuo 可以可靠地评估、确认、恢复到用户确认的网络基线。

### 1.2 范围

本规格覆盖以下产品行为：

1. 安装后网络评估与初始状态快照
2. 用户确认基线（baseline）的交互语义
3. 基线状态项清单、分类、保存、对比与恢复边界
4. 停止服务恢复流程
5. Proxy Guard 异常检测与恢复
6. 工具重启后的未完成恢复动作续跑
7. 本地审计边界
8. 失败解释最小语义
9. Windows 与 WSL/Linux 协同环境的只读评估边界

### 1.3 术语

| 术语 | 定义 |
|------|------|
| 初始状态 | GoGuo 首次运行前本机网络状态的只读快照，用于诊断和人工恢复参考 |
| baseline（基线） | 经过 GoGuo 评估、必要调整和用户确认后的可用网络状态，是后续自动恢复的目标 |
| 状态项 | GoGuo 采集、对比和恢复的网络配置单元（如 hosts 条目、系统代理设置、DNS 缓存等） |
| 可恢复项 | 已验证可被 GoGuo 自动恢复到 baseline 值的状态项 |
| 可检测不可恢复项 | 可被检测到偏离 baseline 但当前无法自动恢复的状态项 |
| 暂不覆盖项 | 本轮不在 baseline 采集和恢复范围内的状态项 |
| Proxy Guard | 持续监控 GoGuo 相关进程、端口和系统代理一致性的守护机制 |
| 续跑 | 工具重启后优先处理上次未完成的恢复动作 |

### 1.4 目标用户

| 用户 | 本规格关注点 |
|------|-------------|
| 普通办公用户 | baseline 确认流程、恢复结果、失败提示可理解性 |
| PC 端开发者/知识工作者 | WSL/Linux 评估、开发工具链网络状态覆盖 |
| 进阶用户/运维调试用户 | 审计记录可读性、手动接管能力 |

### 1.5 成功标准

| # | 标准 | 验证方式 |
|---|------|----------|
| SC-1 | 可恢复项在停止服务后 100% 回到 baseline 值 | 停止服务后逐项对比当前状态与 baseline |
| SC-2 | 不可恢复项 100% 明确提示原因和建议动作 | 遍历不可恢复项场景，检查提示内容 |
| SC-3 | Proxy Guard 异常场景不能让系统代理长期指向死服务 | 注入异常后检查系统代理恢复时间和结果 |
| SC-4 | 工具重启后必须先处理未完成恢复动作，再进入后续自动操作 | 在恢复流程中强制终止工具，重启后检查行为序列 |
| SC-5 | 非目标直连站点在停止恢复后访问成功率不低于 baseline | 对比停止前后非目标站点可达性 |
| SC-6 | 本地审计只记录系统改动和恢复动作，不记录用户访问内容明细 | 审查审计内容 |

## 2. 功能需求

### 2.1 安装后网络评估与初始状态快照

#### FR-2.1.1 初始状态快照采集

GoGuo 首次运行时，在修改任何网络配置前，必须采集当前网络状态作为初始状态快照。

**采集范围（Windows）**:

| 状态项 | 采集内容 | 分类 |
|--------|----------|------|
| Windows hosts 文件 | `%SystemRoot%\System32\drivers\etc\hosts` 全文内容与哈希 | 可恢复项 |
| 系统代理设置 | 注册表 `HKCU\...\Internet Settings` 中的 ProxyEnable、ProxyServer、ProxyOverride | 可恢复项 |
| PAC 设置 | 注册表 `AutoConfigURL` 值 | 可恢复项 |
| WinHTTP 代理 | `netsh winhttp show proxy` 输出 | 可检测不可恢复项 |
| DNS 缓存 | `ipconfig /displaydns` 输出 | 可检测不可恢复项 |
| DNS 服务器配置 | 网络适配器首选/备用 DNS 服务器地址 | 可检测不可恢复项 |
| 代理进程 | 与常见代理客户端（mihomo 等）相关的运行进程列表、监听端口 | 可检测不可恢复项 |
| TUN 状态 | TUN 适配器启用状态与配置 | 可检测不可恢复项 |
| WSL2 网络模式 | WSL2 网络模式（NAT / 镜像） | 可检测不可恢复项 |
| 防火墙规则 | GoGuo 相关入站/出站规则 | 暂不覆盖项 |

**采集范围（WSL/Linux）**:

| 状态项 | 采集内容 | 分类 |
|--------|----------|------|
| WSL 代理环境变量 | `http_proxy`、`https_proxy`、`no_proxy`、`HTTP_PROXY`、`HTTPS_PROXY`、`NO_PROXY` | 可恢复项 |
| WSL Git 代理配置 | `http.proxy`、`https.proxy` | 可恢复项 |
| WSL `/etc/resolv.conf` | DNS 配置内容 | 可恢复项 |
| WSL `/etc/environment` | 系统级代理环境变量 | 可恢复项 |
| WSL shell 配置代理 | `.bashrc`/`.zshrc`/`.profile` 中的代理相关设置 | 可检测不可恢复项 |
| WSL 代理感知状态 | 通过 `curl`/`wget` 测试的当前网络可达性摘要 | 可检测不可恢复项 |
| WSL2 网络模式 | WSL2 网络模式（NAT / 镜像） | 可检测不可恢复项 |

> **注**: 状态项分类基于技术可恢复性（能否可靠读写），不按平台区分。同一类型的状态项（如代理环境变量）无论在哪个平台，只要技术可读写，均归类为"可恢复项"。本轮是否实际执行恢复由 CON-1 约束决定，与分类无关。自动配置接管（按目标站点规则自动调整网络配置）属于 OPP-004。

**要求**:
- FR-2.1.1-R1: 采集必须在 GoGuo 修改任何网络配置之前完成
- FR-2.1.1-R2: 采集结果保存为只读快照，后续不可被 GoGuo 自动修改
- FR-2.1.1-R3: 快照必须包含采集时间戳和采集环境信息（OS 版本、WSL 版本等）
- FR-2.1.1-R4: 若某状态项采集失败，必须记录失败原因并标记该状态项为"采集失败"，不阻塞后续流程

#### FR-2.1.2 网络评估

采集初始状态后，GoGuo 必须执行基础网络可达性评估。

**要求**:
- FR-2.1.2-R1: 评估必须包含对常见直连站点（如百度、必应中国）的可达性测试
- FR-2.1.2-R2: 评估必须包含对用户预期目标站点的可达性测试（若用户已配置）
- FR-2.1.2-R3: 评估结果必须与初始状态快照一起保存
- FR-2.1.2-R4: 评估过程不得修改任何系统网络配置

### 2.2 用户确认基线（baseline）

#### FR-2.2.1 基线形成

初始状态快照和评估完成后，GoGuo 必须协助用户形成确认的 baseline。

**要求**:
- FR-2.2.1-R1: baseline 默认等于初始状态快照的当前值（即"当前状态可用，确认即可"路径）
- FR-2.2.1-R2: 若用户需要 GoGuo 进行网络调整（如设置系统代理），调整后必须重新采集受影响的状态项，更新 baseline 候选
- FR-2.2.1-R3: baseline 必须包含所有已采集状态项的值，包括"可恢复项"和"可检测不可恢复项"
- FR-2.2.1-R4: baseline 一旦确认，保存为只读记录；后续修改必须产生新的 baseline 版本

#### FR-2.2.2 用户确认交互

**要求**:
- FR-2.2.2-R1: 确认前必须向用户展示 baseline 包含的状态项摘要，区分"可恢复项"和"可检测不可恢复项"
- FR-2.2.2-R2: 确认文案必须明确说明 baseline 是"用户确认的可用状态"，而非"系统出厂状态"
- FR-2.2.2-R3: 必须提供二次确认机制（如确认对话框或命令行确认提示）
- FR-2.2.2-R4: 用户确认动作必须记入本地审计

### 2.3 基线状态项管理

#### FR-2.3.1 状态项分类

**要求**:
- FR-2.3.1-R1: 每个状态项必须被分类为"可恢复项"、"可检测不可恢复项"或"暂不覆盖项"
- FR-2.3.1-R2: 分类必须包含分类理由
- FR-2.3.1-R3: 分类结果必须持久化保存，可被后续恢复流程引用

#### FR-2.3.2 状态项保存与对比

**要求**:
- FR-2.3.2-R1: baseline 中每个状态项必须保存其当前值和采集时间
- FR-2.3.2-R2: 对比操作必须逐项比较当前值与 baseline 值，输出差异清单
- FR-2.3.2-R3: 差异清单必须区分"偏离 baseline"和"一致"两种状态

#### FR-2.3.3 恢复边界

**要求**:
- FR-2.3.3-R1: 仅"可恢复项"参与自动恢复；"可检测不可恢复项"仅输出提示
- FR-2.3.3-R2: 自动恢复前必须先与 baseline 对比，仅恢复有差异的项
- FR-2.3.3-R3: 恢复后必须立即验证，确认当前值与 baseline 一致
- FR-2.3.3-R4: 恢复验证失败时，不得静默跳过，必须记录并提示用户

### 2.4 停止服务恢复流程

#### FR-2.4.1 停止触发

**要求**:
- FR-2.4.1-R1: 用户执行停止操作时，必须触发 baseline 差异检查和恢复流程
- FR-2.4.1-R2: 停止操作不能只终止进程，必须执行完整的恢复检查
- FR-2.4.1-R3: 若停止过程中恢复流程失败或被中断，必须将未完成恢复动作持久化为恢复任务记录

#### FR-2.4.2 恢复执行

**要求**:
- FR-2.4.2-R1: 恢复必须按状态项分类执行：可恢复项自动恢复，不可恢复项输出提示
- FR-2.4.2-R2: 恢复执行顺序应按风险从低到高排列（如先恢复系统代理，再恢复 PAC 设置，最后恢复 hosts）
- FR-2.4.2-R3: 恢复过程中每个状态项的恢复结果（成功/失败/跳过）必须记入审计
- FR-2.4.2-R4: 全部恢复完成后，必须执行验证：逐项对比当前值与 baseline
- FR-2.4.2-R5: 验证结果必须向用户展示，区分"已恢复"和"需要手动处理"两类

#### FR-2.4.3 非目标直连站点验证

**要求**:
- FR-2.4.3-R1: 停止恢复后，必须验证非目标直连站点的可达性不低于 baseline 记录的水平
- FR-2.4.3-R2: 若非目标直连站点可达性低于 baseline，必须在恢复结果中明确提示

### 2.5 Proxy Guard

#### FR-2.5.1 监控范围

Proxy Guard 必须持续监控以下组件的一致性：

| 监控对象 | 监控内容 | 预期行为 |
|----------|----------|----------|
| GoGuo 客户端进程 | 进程存活状态 | 存活 |
| 监控服务（MonitorServer） | 进程存活 + 健康 API 响应 | 存活且响应正常 |
| 代理核心（mihomo 等） | 进程存活 + 端口监听 + 健康 API | 存活且端口正常 |
| 相关子进程 | mihomo 核心、MonitorServer、DNS 辅助进程、健康检查辅助进程、历史运行残留进程 | 无残留 |
| 系统代理设置 | ProxyEnable/ProxyServer 值 | 与当前模式一致 |
| 代理端口 | 监听端口可达性 | 可达 |

#### FR-2.5.2 异常检测与响应

**要求**:
- FR-2.5.2-R1: 检测到代理核心异常（崩溃、端口不可达、健康 API 异常）时，必须在系统代理残留被清除前阻止新请求通过死代理
- FR-2.5.2-R2: 检测到 GoGuo 客户端自身异常时，必须尝试触发恢复路径
- FR-2.5.2-R3: 检测到子进程残留时，必须清理并记录
- FR-2.5.2-R4: 检测到系统代理指向不可达服务时，必须立即将系统代理恢复到 baseline 值
- FR-2.5.2-R5: 异常处理结果必须记入审计

#### FR-2.5.3 恢复策略

**要求**:
- FR-2.5.3-R1: 代理核心崩溃时，应尝试重启代理核心
- FR-2.5.3-R2: 重启失败时，应将系统代理恢复到 baseline 并提示用户
- FR-2.5.3-R3: 连续重启失败超过阈值时，应停止尝试并提示用户手动处理
- FR-2.5.3-R4: 阈值应可配置，默认值不超过 3 次

### 2.6 工具重启后续跑

#### FR-2.6.1 未完成恢复任务检测

**要求**:
- FR-2.6.1-R1: 工具启动时必须首先检查是否存在未完成的恢复任务记录
- FR-2.6.1-R2: 恢复任务记录必须持久化存储（非仅内存），在工具崩溃或被强制终止后仍可读取
- FR-2.6.1-R3: 恢复任务记录必须包含：未完成的状态项列表、已尝试的动作、失败原因

#### FR-2.6.2 续跑执行

**要求**:
- FR-2.6.2-R1: 若存在未完成恢复任务，必须先执行续跑，再进入后续自动操作
- FR-2.6.2-R2: 续跑过程中不允许启动新的网络修改操作（如切换模式、应用规则）
- FR-2.6.2-R3: 续跑完成后必须执行验证，确认恢复结果
- FR-2.6.2-R4: 续跑结果必须向用户展示

### 2.7 本地审计

#### FR-2.7.1 审计范围

**必须记录**:
- FR-2.7.1-R1: 每次 baseline 采集和确认操作
- FR-2.7.1-R2: 每次系统网络配置修改（修改前值、修改后值、修改原因）
- FR-2.7.1-R3: 每次恢复动作（目标状态项、恢复结果、成功/失败原因）
- FR-2.7.1-R4: Proxy Guard 异常检测和处理结果
- FR-2.7.1-R5: 用户确认操作

**禁止记录**:
- FR-2.7.1-R6: 用户访问内容明细（URL、请求内容、响应内容）
- FR-2.7.1-R7: 用户浏览历史
- FR-2.7.1-R8: 用户流量内容

#### FR-2.7.2 审计格式

**要求**:
- FR-2.7.2-R1: 每条审计记录必须包含：时间戳、操作类型、操作对象、操作结果、操作原因
- FR-2.7.2-R2: 审计记录存储在 GoGuo 安装根目录下的统一数据目录
- FR-2.7.2-R3: 审计记录格式应支持基本的结构化查询（如按时间范围、操作类型筛选）

### 2.8 失败解释

#### FR-2.8.1 失败提示要素

每次恢复失败或异常处理失败时，提示必须包含以下五要素：

| 要素 | 说明 | 示例 |
|------|------|------|
| 原因 | 当前失败的技术原因 | "系统代理设置恢复失败：注册表写入被拒绝" |
| 已尝试动作 | GoGuo 已执行的恢复尝试 | "已尝试 3 次恢复系统代理" |
| 尝试次数 | 累计尝试次数 | "3/3" |
| 建议动作 | 用户下一步应执行的操作 | "请检查是否有其他软件锁定了系统代理设置，或尝试以管理员身份运行 GoGuo" |
| 是否需要手动处理 | 明确告诉用户是否需要人工介入 | "需要手动处理" |

**要求**:
- FR-2.8.1-R1: 五要素必须完整，不得省略任何一项
- FR-2.8.1-R2: 提示文案应面向非技术用户可理解，避免暴露原始错误码（但审计中保留原始错误码）
- FR-2.8.1-R3: 提示中如包含建议动作，建议动作必须是用户可执行的（如"以管理员身份运行"而非"检查权限"）

### 2.9 Windows 与 WSL/Linux 协同环境只读评估

#### FR-2.9.1 部署组合识别

**要求**:
- FR-2.9.1-R1: GoGuo 必须识别当前运行环境属于以下部署组合之一：
  - 仅 Windows 部署，WSL 未安装或未运行
  - 仅 WSL/Linux 部署（GoGuo 运行在 WSL 内）
