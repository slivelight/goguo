## 1. 概述

### 1.1 目的

本文档定义 GoGuo 用户交互界面的需求规格。作为独立桌面应用，承载 Feature 001（baseline/restore）、Feature 002（WSL/Linux 配置）和 Feature 003（目标站点规则）定义的全部交互语义，实现从安装→配置→监控→诊断→恢复的完整用户操作闭环。

### 1.2 范围

1. 服务状态展示与控制
2. 目标站点管理
3. 规则预览与确认
4. 状态监控与诊断
5. 通知机制
6. 首次引导流程（含手工调整引导）
7. Windows 与 WSL/Linux 两侧 UI 可访问性

### 1.3 术语

| 术语 | 定义 |
|------|------|
| 桌面应用 | GoGuo 的独立跨平台桌面应用程序，非 Web UI 或浏览器面板 |
| 交互语义 | Feature 001/002/003 规格中定义的、需要 UI 承载的用户交互行为 |
| 主路径 | 用户完成核心操作的最短路径（战略发现要求不超过 2 步） |

### 1.4 目标用户

| 用户 | 本规格关注点 |
|------|-------------|
| 普通办公用户 | 一眼看到状态、预设模板快速配置、无需理解技术细节 |
| PC 端开发者 | 可视化管理站点、实时查看诊断结果、规则预览 |
| 进阶用户 | 规则覆盖和自定义能力 |

### 1.5 成功标准

| # | 标准 | 验证方式 |
|---|------|----------|
| SC-1 | 主路径操作不超过 2 步 | 核心操作路径步骤计数验证 |
| SC-2 | 状态显示与后端实际状态一致 | UI 显示与 API 返回数据对比 |
| SC-3 | 失败提示可触发手动处理 | 注入失败场景，验证提示引导用户完成处理 |
| SC-4 | 全部通知（规则回退、恢复进度、审计变更）可在 UI 中查看 | 通知场景触发后检查 UI 通知区 |
| SC-5 | Windows 和 WSL/Linux 两侧均可访问且操作界面一致 | 两侧分别运行，对比 UI 一致性 |

## 2. 功能需求

### 2.1 应用框架

#### FR-2.1.1 跨平台桌面应用

**要求**:
- FR-2.1.1-R1: 必须以独立桌面应用形态运行，不是 Web UI 或浏览器面板
- FR-2.1.1-R2: 必须在 Windows 和 WSL/Linux 两侧均可运行
- FR-2.1.1-R3: 两侧必须保持一致的程序操作界面（布局、交互方式、功能入口一致）
- FR-2.1.1-R4: UI 全部数据必须来源于本地 API，不得发起远程请求
- FR-2.1.1-R5: 应用必须能在后端未运行时启动并展示离线状态（参见 OP-1）
- FR-2.1.1-R6: 当前阶段不提供用户登录/鉴权管理，启动用户即以管理员身份运行；此能力延迟到 OPP-005

### 2.2 首次引导流程

#### FR-2.2.1 Baseline 评估与确认

**要求**:
- FR-2.2.1-R1: 首次启动时，必须引导用户进入 baseline 评估流程
- FR-2.2.1-R2: 必须展示当前网络状态采集结果（按 Feature 001 定义的状态项分类展示）
- FR-2.2.1-R3: 用户确认 baseline 前，必须提供 step by step 手工调整引导：展示当前状态中不理想的项，逐项给出调整建议，包括可执行的命令行命令（如可一键复制）或系统页面操作指引（如"打开 设置 → 网络和 Internet → 代理"），指导用户手工调整系统网络配置<!-- [TODO] id:02;status:close;date:2026-05-12T09:00  提供step by step方式指导用户手工调整系统网络配置，提供可供执行的命令行命令，或系统页面操作指引；任务处理结果：已修订 FR-2.2.1-R3，增加 step by step 方式、可执行命令（支持一键复制）和系统页面操作指引。-->
- FR-2.2.1-R4: 手工调整后，用户可触发重新采集→重新评估→重新确认 baseline 的完整流程
- FR-2.2.1-R5: Baseline 确认操作必须遵循 Feature 001 的二次确认机制

#### FR-2.2.2 部署模式选择

**要求**:
- FR-2.2.2-R1: 首次引导时，必须提供部署模式选择（协同/仅 Windows/仅 WSL，按 Feature 002 定义）
- FR-2.2.2-R2: 选择后必须展示该模式的含义和影响说明
- FR-2.2.2-R3: 部署模式可在后续设置中更改（仅影响运行时行为，不涉及重新安装）

### 2.3 服务状态展示与控制

#### FR-2.3.1 服务控制

**要求**:
- FR-2.3.1-R1: 必须提供 GoGuo 服务的启动/停止控制
- FR-2.3.1-R2: 服务启动/停止属于敏感操作，必须遵循 Feature 001 的二次确认机制
- FR-2.3.1-R3: 服务停止时必须展示 baseline 恢复进度（按 Feature 001 定义）

#### FR-2.3.2 状态展示

**要求**:
- FR-2.3.2-R1: 必须展示 GoGuo 当前运行状态（运行中/已停止/异常）
- FR-2.3.2-R2: 必须展示当前 baseline 状态（已确认/未确认/已偏离）
- FR-2.3.2-R3: 必须展示 Windows 与 WSL/Linux 两侧的配置状态（按 Feature 002 定义的部署模式）
- FR-2.3.2-R4: 当两侧配置存在差异时，必须提供差异提示

### 2.4 目标站点管理

#### FR-2.4.1 站点添加与删除

**要求**:
- FR-2.4.1-R1: 必须支持通过站点标识（如 `github`）添加目标站点
- FR-2.4.1-R2: 必须支持通过域名（如 `github.com`）添加目标站点
- FR-2.4.1-R3: 输入站点标识后，必须展示站点定义展开的关联域名列表供用户确认（按 Feature 003 FR-2.1.1-R3）
- FR-2.4.1-R4: 必须支持删除已添加的目标站点
- FR-2.4.1-R5: 站点列表变更必须在 UI 中即时反映

#### FR-2.4.2 预设模板选择

**要求**:
- FR-2.4.2-R1: 必须提供预设站点模板选择（开发者套装/办公套装，按 Feature 003 FR-2.1.2-R3 定义）
- FR-2.4.2-R2: 选择模板后，必须展示模板包含的站点列表供用户确认
- FR-2.4.2-R3: 用户可在模板基础上增删站点

### 2.5 规则预览与确认

#### FR-2.5.1 规则预览

**要求**:
- FR-2.5.1-R1: 规则应用前，必须提供规则预览，展示即将生效的规则列表
- FR-2.5.1-R2: 预览中必须标注每条规则对应的站点名称和策略（PROXY/DIRECT）
- FR-2.5.1-R3: 规则预览不得泄露用户站点列表到远程（本地渲染）
- FR-2.5.1-R4: 批量规则预览（如基于预设模板）时，应采用与单站点规则预览一致的交互风格：按站点分组展示，每组可展开查看关联域名规则，支持折叠/展开切换
<!-- [?] id:04;status:close;date:2026-05-12T09:00  根据预设模板来批量设置目标站点时，其代理规则将会批量被应用，此时如何批量预览即将生效的规则需要更抽象/直观的用户交互设计，与单个目标站点的规则预览尽量保持一致的风格；任务处理结果：新增 FR-2.5.1-R4，要求批量预览采用与单站点一致的交互风格（按站点分组、可展开/折叠），保持设计一致性。-->
#### FR-2.5.2 规则确认

**要求**:
- FR-2.5.2-R1: 规则应用必须经用户确认，遵循 Feature 001 的二次确认机制
- FR-2.5.2-R2: 规则应用后，必须展示应用结果（成功/失败/回退）
- FR-2.5.2-R3: 若规则回退，必须展示回退原因（按 Feature 003 FR-2.2.2-R3）

#### FR-2.5.3 规则覆盖

**要求**:
- FR-2.5.3-R1: 进阶用户可对自动生成的规则进行覆盖（按 Feature 003 FR-2.4.2）
- FR-2.5.3-R2: 覆盖操作必须在 UI 中明确标识为用户自定义规则

### 2.6 状态监控与诊断

#### FR-2.6.1 目标站点可达性

**要求**:
- FR-2.6.1-R1: 必须实时展示每个目标站点的可达性状态（可达/不可达/探测中）
- FR-2.6.1-R2: 必须展示可达性探测的响应时间
- FR-2.6.1-R3: 不可达站点必须在视觉上突出显示

#### FR-2.6.2 诊断提示

**要求**:
- FR-2.6.2-R1: 目标站点不可达时，必须展示五要素诊断提示（原因、已尝试动作、尝试次数、建议动作、是否需要手动处理，按 Feature 003 FR-2.3.2-R2）
- FR-2.6.2-R2: 诊断提示中必须包含用户可直接执行的建议动作
- FR-2.6.2-R3: 恢复进度（如正在切换节点）必须实时展示

#### FR-2.6.3 节点池状态

**要求**:
- FR-2.6.3-R1: 必须展示代理节点池的整体状态（可用节点数/总数）
- FR-2.6.3-R2: 必须展示当前使用的代理节点
- FR-2.6.3-R3: 必须展示每个代理节点的元数据信息：节点名称、入池时间、当前可用状态、最近可用检测时间、可用检测方式
<!-- [?] id:05;status:close;date:2026-05-12T09:00  FR-2.6.3-R1中，还包括每个代理节点的名称，入池时间，是否可用，可用检测时间,可用检测方式等元数据信息；任务处理结果：新增 FR-2.6.3-R3，明确每个代理节点需展示的元数据字段（名称、入池时间、可用状态、检测时间、检测方式）。-->
### 2.7 通知机制

#### FR-2.7.1 应用内通知

**要求**:
- FR-2.7.1-R1: 必须提供应用内通知区域，展示所有通知
- FR-2.7.1-R2: 以下事件必须产生通知：
  - 规则回退（含回退原因）
  - 恢复动作（含恢复进度）
  - 审计变更（按 Feature 001 审计流）
  - 节点池变更（节点移除/恢复）
- FR-2.7.1-R3: 通知必须包含时间戳
- FR-2.7.1-R4: 通知不得产生用户不可控的副作用（如阻塞操作、弹窗刷屏）

#### FR-2.7.2 系统通知

**要求**:
- FR-2.7.2-R1: 重要通知（规则回退、恢复失败）应同时通过系统通知机制推送
- FR-2.7.2-R2: 系统通知不可用时，应用内通知作为降级方案，不得丢失通知内容
